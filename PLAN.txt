2. BUNDLE-A.JS LAZY-LOAD (UX, the remaining UX win)
   ─────────────────────────────────────────────────
   bundle-a.js is 11.7 MB on every cold WebView boot. books.js (NKJV) alone
   is 6.9 MB = 60% of the cost. Mid-range Android: ~3-5s parse before any
   UI paints.

   Analysis + implementation strategy: BUNDLE-LAZY-LOAD-PLAN.md (Q8.0,
   commit f7dff63). The plan doc has the full design — sufficient detail
   that a follow-up session can pick up cleanly.

   Why deferred from this session: BOOKS is read at first paint by
   ScripturesHome (tile counts) + Settings (translation list) + 8 hooks.
   A clean lazy-load needs either guards-at-every-access-site or a
   skeleton-state design that fills in once the corpus loads. The smoke
   matrix is ~30 screens; rushing it would ship subtle regressions or
   miss the actual UX win.

   Exit criteria: bundle-a.js < 4 MB, cold-boot to first paint < 2 sec on
   mid-range Android, every BOOKS access site guarded or awaited, full
   smoke walk passing identically.

INFRA / TOOLING ITEMS (one-liners — open small, ship fast)
───────────────────────────────────────────────────────────────────────────────

- lint:globals AUTO-REGEN in pre-commit hook: today the hook rebuilds bundles
  but does NOT run `npm run lint:globals`. If a dev adds a new global export,
  the committed globals file lags reality until CI regenerates it. Mirror
  the bundle-rebuild logic: regenerate + restage globals on any source change.

- LINT-STAGED auto-fix mode: currently `eslint --max-warnings 0` (no --fix).
  If devs want auto-fix on `--fix`-able warnings (unused-vars `_` prefix,
  whitespace), add a second config entry. Low priority — auto-fix can hide
  intentional patterns.

- `tsc --noEmit` IN CI ✅ DONE (Q4.1, `001747b`) — runs between lint and
  build with zero-tolerance. Pre-commit Step 3 mirrors the gate.
