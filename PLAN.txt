PWA MIGRATION — CROSS-PLATFORM (Android + PC)
═══════════════════════════════════════════════════════════════════════════════

Goal: VOTReader runs as a native Android APK (existing) AND as an installable
PWA on desktop (Chrome/Edge/Firefox). One JS codebase, two shells. All data
stays local on both platforms. No server, no credentials, no API keys.

Rationale — the inversion:

  The architectural review surfaced 18 critiques. The deepest one — "WebView
  fighting native UX" — is the meta-critique behind most of the others.
  Going cross-platform doesn't just ADD a platform. It INVERTS that critique.
  The 135+ modules of battle-tested web code stop being a limitation locked
  inside a single-platform Android shell and become the core asset. PWA
  validates the foundation rather than rebuilding it.

  Concretely, every strength W0-W8 leans on is already in place:
    - AndroidBridge call sites are already guarded with `if (window.AndroidBridge)`
      — desktop browser mostly works today (W1 is formalization, not invention).
    - N1's native extraction (BridgeHost / AppInterface / JsBridge /
      MainViewModel / NativeAudioRecorder / StorageManager) already demonstrates
      the shell-vs-app separation that PlatformBridge will mirror on the JS side.
    - The 6-step pre-commit gate + 476 vitest + 134 JUnit + JaCoCo 0.85 floor
      means changes are inspectable per-commit. The safety net is built;
      this work uses it rather than rebuilding it.
    - 1.03 MB critical-path bundle is small enough to cache as the PWA
      install-time shell (W3).

  One codebase, two shells, all data local. No server, no credentials, no API
  keys, no runtime network egress. The PWA isn't a port — it's revealing what
  was already there.

  W0-W6 makes the app work everywhere. W7-W8 closes every remaining critique.
  When W8 closes, the 18-item traceability matrix at the bottom of this file
  is clean — zero gaps.

  Mandate: not just executing the plan. Stay alert to anything broken or
  missable as we touch each file. Improve as we go. Creative & professional.
  Every commit ships to production.

Current state (pre-migration):
  - The JS app ALREADY mostly works in a desktop browser (React, stores,
    corpus, rendering all load). AndroidBridge calls silently no-op because
    every call site is already guarded: `if (window.AndroidBridge) {...}`.
  - What DOESN'T work on desktop: file import/export, audio recording,
    native screenshots, back-button navigation, fullscreen toggle, zoom,
    haptics, status bar, keep-screen-on.
  - Font files are declared in CSS @font-face but DO NOT EXIST in the repo
    (fonts/ directory is empty). This is a pre-existing visual bug on any
    platform — system fallback fonts render instead of Cinzel/Garamond.
  - localStorage persistence works on both but has quota ceiling (~5-10 MB)
    and no transactional writes.
  - No PWA infrastructure exists: no manifest.json, no service worker, no
    offline fallback, no PWA meta tags.

Sequencing rationale:
  W0 → W1 → W2 → W3 → W4 → W5 → W6

  W0 (prerequisites) unblocks everything: fonts + owed device verification.
  W1 (platform bridge) makes desktop functionally correct — file I/O, audio,
    back-button, screenshots all work via browser APIs.
  W2 (IndexedDB) fixes the persistence ceiling on BOTH platforms — the one
    change with highest user-facing impact.
  W3 (PWA shell) makes it installable + offline on desktop.
  W4 (desktop polish) optimizes the desktop experience after it works.
  W5 (hosting + CI) ships it.
  W6 (cross-platform verification) proves it.

  Each phase is independently shippable. W1 alone makes the app usable on
  desktop. W2 alone improves both platforms. Nothing breaks mid-migration
  because every phase is additive — the Android path continues to work
  unchanged throughout.

W1 — PLATFORM BRIDGE (decouple JS from AndroidBridge)
───────────────────────────────────────────────────────────────────────────────

Goal: every feature that currently requires AndroidBridge works on desktop
via browser-native APIs. The Android path is unchanged — it still uses
the native bridge. Desktop gets its own implementation.

Architecture: a single `platform-bridge.js` module (new file in src/utils/)
exports a PlatformBridge object. At boot, it detects the environment and
wires up the correct implementation. All existing call sites change from
`window.AndroidBridge.foo()` to `PlatformBridge.foo()`.

Detection: `!!(typeof window !== 'undefined' && window.AndroidBridge)` —
truthy check matching the existing pattern at JournalRecordingSheet:214
verbatim. Per [[platformbridge-single-source]], this expression appears
in EXACTLY ONE place: platform-bridge.js. Consumers never check
window.AndroidBridge directly after W1.2.

  W1.5 — Back-button navigation (History API + Escape key)
  ────────────────────────────────────────────────────────
  On Android, back-press routes through `window.handleAndroidBack()`.
  On desktop, wire two triggers:

    1. Keyboard: keydown 'Escape' → handleAndroidBack() — GATED.
       Per [[escape-key-gating]], the listener fires ONLY if:
         - document.fullscreenElement === null  (not in fullscreen)
         - No app-managed modal/sheet open
       If either condition false: do NOT preventDefault — let the
       browser exit fullscreen / close its own dialog. Swallowing
       Escape would break the Garden full-screen view + every
       open-sheet close-on-escape pattern.

       Per [[centralized-modal-signal]], the "no modal/sheet open"
       check MUST go through a SINGLE centralized signal — NOT a
       multi-source state-check on AppShellSheets + AppShellOverlays +
       screen-local sheet states. Implementation: a useModalRegistry
       hook that exposes register() / unregister() + a stable
       isAnyOpen() ref. Each sheet/overlay calls register() in its
       mount effect and unregister() on cleanup. Counter-based (not
       state-based) so identity churn doesn't drop signals. New sheets
       added later automatically participate via the same register
       pattern — no need to update the Escape handler.

    2. Browser: History API popstate → handleAndroidBack(). The app
       pushes a history entry on each in-app navigation; popstate
       signals the user clicked the browser-native back button.

       Per [[root-of-history-pwa]], popstate handling at the navigation
       ROOT needs special care — a PWA has no "minimize" equivalent
       to Android's behavior when handleAndroidBack returns "false".
       Without a root-of-stack guard, the first back-press past the
       app's home screen EXITS the PWA silently (browser navigates
       away or closes the standalone window). useAndroidBack tracks
       whether the current state IS the root; at root, the popstate
       handler pushes a replacement history entry (history.pushState)
       so the next back-press lands inside the app, not outside.
       Optionally show a brief "Press back again to exit" toast on
       first root-back to match the Android UX pattern.

  The existing handleAndroidBack logic is platform-agnostic (reads JS
  state, returns "true"/"false"). The trigger source changes; the
  return-handling for "false" diverges by platform (Android: minimize;
  Web: stay at root). Register listener in useAndroidBack.js (already
  owns this domain). Tests:
    - (fullscreen, sheet-open, clear-state) Escape → only clear-state dispatches
    - popstate at root → stays in app (history entry pushed)
    - popstate mid-stack → handleAndroidBack runs normally

W2 — STORAGE HARDENING (persistence + scale + safety)
───────────────────────────────────────────────────────────────────────────────

Goal: every byte of user data — structured metadata AND binary media —
is durably stored, protected from eviction, scalable to GB on every
platform, safe on budget devices, and fully round-tripped through
export/import. When W2 closes, the storage layer is production-grade
on Android WebView, Chrome, Edge, Firefox, and Safari (including iOS

Architecture — what exists, what changes:

    - 7 stores with useSyncExternalStore reactivity.

  CHANGES IN W2:

    Baked-in constraints:
    1. Safari evicts ALL storage after 7 days of no interaction unless
       the user has added the PWA to Home Screen. Home-screen PWAs are
       exempt. The Safari install prompt (W3) is a DATA PRESERVATION
       requirement, not a nice-to-have.
    2. Budget Android with <500 MB free disk gets quota as low as a few
       MB. The app must detect this and warn, not crash.
    3. Firefox requires persist() from a user gesture (click handler).
       Silent failure otherwise.

       from Safari tabs does NOT carry over. Users must install BEFORE
       creating data, or use export/import to bridge.

    6. For playback/display of large blobs: ALWAYS use
       URL.createObjectURL(blob) or blob.stream(). NEVER use

  W2.2 — Write-through cache layer (modify cached-store.js)
  ──────────────────────────────────────────────────────────
  Extend CachedStore to support an async backing store:

    _load()    → read from in-memory cache (sync, fast)
                  if cache is cold, return default (empty)
    _save()    → write to in-memory cache (sync, immediate)
                  ALSO write to IDB (async, with error handling —
                  every write wraps in QuotaExceededError catch →
                  StorageHealth.onWriteFailure)
    _hydrate() → async one-time load from IDB into memory
                  called at boot, before React renders

  The hydration gate: App() renders a loading screen until all stores
  report hydrated. This is the ONLY async boundary — after hydration,
  every read is sync from memory. Zero render-path jank.

  Boot sequence:
    1. index.html inline script reads `vot-state` from localStorage
       (sync) for theme class — KEEP THIS as the fast-path shim. The
       localStorage copy is updated on every _save() as a write-behind
       mirror for the next cold boot's theme shim.
    2. React mounts, App() calls hydrateAllStores() (async).
    3. Each store opens IDB, reads data, populates in-memory cache.
    4. Promise.all resolves. App() sets hydrated=true, renders.
    5. All subsequent reads are sync from memory.

  Hydration timeout — data-loss vector (per [[w2-hydration-data-loss]]):
  if IDB open takes >3 seconds, the naive impl renders with an empty
  cache + a background retry. THIS IS DANGEROUS: user starts interacting
  with the empty cache, writes data, then the retry resolves and the
  loaded IDB data STOMPS the user's writes. Silently lost data.

  Required design — degraded mode + write-queue:

    State machine per store: 'pending' | 'loaded' | 'degraded'.

    'pending' (boot → hydration in flight):
       Reads return default values.
       Writes go to a per-store WRITE QUEUE (not the cache yet).
       UI shows a brief "Loading..." indicator on the affected surfaces
       so user knows their action is staged but not yet persistent.

    'loaded' (hydration succeeded within timeout):
       Drain write queue → apply to cache (each enqueued write wins
       over the loaded IDB value at the same key).
       Flush cache → IDB.
       Normal write-through from this point.

    'degraded' (hydration timeout, e.g. 3s elapsed):
       Mark store degraded.
       Drain write queue → apply to cache as authoritative.
       Treat cache as truth from this point.
       Continue background IDB load attempts; on eventual success,
       MERGE — cache wins on every conflicting key.
       Surface a brief UI warning: "Storage temporarily unavailable;
       your latest changes are kept in memory and will save when
       storage recovers."

    Background retry on persistent failure → store stays 'degraded'
    forever (until app reload). Cache is the only source of truth.
    Per [[user-data-paramount]], silent stomping is unacceptable;
    a degraded-but-functional app is better than data loss.

  Tests (must exist in W2.2 vitest):
    1. Hydration <100ms → 'loaded' state → write-through works normally
    2. Hydration >3s + user writes during pending → user writes survive
       when hydration eventually resolves (NOT stomped)
    3. Hydration NEVER completes (corrupted DB / permission denied) →
       store stays 'degraded' but app remains usable; user-facing
       warning surfaced
    4. Multiple stores hydrate concurrently — some succeed, some time
       out. Each store's state tracked independently

  W2.4 — One-time localStorage → IndexedDB migration
  ───────────────────────────────────────────────────
  On first boot post-migration:
    1. Open IDB. Check for "migrated-v1" key in a "meta" store.
    2. If absent: read every vot-* localStorage key, write to IDB.
    3. Set "migrated-v1" in IDB meta store.
    4. Clear migrated keys from localStorage (free quota) EXCEPT:
       vot-state stays (fast theme shim). Gets a reduced copy (theme
       + essential boot props only) to minimize localStorage usage.
    5. Log migration timing.

  Idempotent: if migration is interrupted, next boot retries. The
  "migrated-v1" flag is set LAST, after all writes succeed.

  VOTReader in Safari tab and then installs the PWA, the installed PWA

  localStorage, finds nothing, sets the flag. User sees blank app.

  and surface the iOS PWA welcome card (W2.7 Scenario 4).

  W2.7 — Storage Health system (dynamic, platform-aware user warnings)
  ────────────────────────────────────────────────────────────────────
  The app reads the storage situation continuously and communicates
  risks to the user in plain language — before they lose data, not
  after. No jargon. Every warning tells the user what to DO. Warnings
  are invisible when everything is fine.

  Architecture: StorageHealth module (src/utils/storage-health.js) owns
  detection, assessment, and state. UI components subscribe to it.

  Detection engine — StorageHealth.assess() runs:
    1. Platform detection (one-time, cached):
       Android WebView / Safari tab / Safari PWA / Firefox / Chrome /
       private browsing (probe via IDB open + quota check)
    2. Quota snapshot (async): navigator.storage.estimate()
    3. Persistence check: navigator.storage.persisted()
    4. Derived health tier:
       HEALTHY  — <50% used, persisted, no platform risks
       CAUTION  — 50-80% used, OR not persisted, OR Safari tab
       WARNING  — 80-95% used, OR quota < 100 MB
       CRITICAL — >95% used, OR quota < 50 MB, OR private browsing
       READONLY — active write failure this session (QuotaExceededError)
    5. Risk flags (zero or more): safari-7day, ios-pwa-isolate,
       low-quota, critical-quota, not-persisted, private-mode,
       write-failed, quota-declining

  StorageHealth API:
    assess()                → Promise<StorageHealthReport>
    checkBeforeWrite(bytes) → { ok, warning? }
    onWriteFailure(error)   → updates tier to READONLY
    onWriteSuccess()        → clears READONLY if quota recovered
    subscribe(callback)     → unsubscribe function
    getReport()             → last cached report (sync, for render)
    requestPersistence()    → Promise<boolean>

  When to check:
    1. Boot (after hydration)
    2. After every meaningful data creation (journal, annotation, etc.)
       — only re-assess if tier >= CAUTION
    3. Periodic: every 5 min while app is active (disk space changes)
    4. After any write failure: immediate re-assess
    5. On resume from background (visibilitychange): if >60s stale

  W7.2 — raw() immutability (store mutation safety)
  ─────────────────────────────────────────────────
  Problem: `CachedStore.raw()` returns the live `_cache` object. A
  consumer can mutate the returned object without calling `_save()`,
  and the change persists in memory but never reaches IndexedDB. This
  is a silent-corruption vector.

  Current state: research found only 1 call site (a test), and zero
  production mutations via raw(). But the API PERMITS it, and new code
  could introduce the bug unknowingly.

  Fix: `raw()` returns `Object.freeze()` of the cache (shallow freeze).
  Consumers that need to mutate must go through the store's named methods
  (add/remove/update/set) which call `_save()` + `_bump()` internally.

  Why Object.freeze, not structuredClone:
    - Freeze is O(1) (just sets a flag on the object), clone is O(n).
    - Annotations store can be 500 KB — cloning on every read is too
      expensive for the render-path hot stores.
    - Freeze catches accidental mutation at dev time (throws in strict
      mode, silent no-op in sloppy mode). Good enough — strict mode is
      default in ES modules.

  Implementation:
    - cached-store.js: change `raw() { return this._load(); }` to
      `raw() { const d = this._load(); Object.freeze(d); return d; }`
    - Stores that return nested objects (annotations has hlKey → Array):
      freeze the top level only. Deep-freeze is too expensive and
      unnecessary — named methods are the write path.
    - Run all 476 vitest tests — any test that mutates via raw() will
      throw, exposing the violation.

  Tests:
    - New vitest case: `raw()` returns frozen object, mutation throws
    - Existing tests pass (proves no production code mutates via raw())

  W7.6 — OPFS upgrade for JournalMediaStore (perf optimization)
  ─────────────────────────────────────────────────────────────
  Problem: JournalMediaStore uses IndexedDB Blob storage (correct,
  GB-scale, works on all platforms). But IDB Blob writes for 100 MB
  payloads take ~850 ms vs ~90 ms with OPFS. For power users with
  hundreds of recordings, the cumulative I/O overhead matters.

  OPFS advantages: 3-4x faster writes, partial range reads via
  `read(buffer, { at: offset })` without loading whole file, no
  structured clone overhead.

  OPFS constraints:
    - Fast sync API (createSyncAccessHandle) ONLY works in a Web Worker
    - Safari has known bugs (WebKit #250495 — data loss on handle create)
    - Firefox: OPFS-only, no full File System Access API
    - Async-only API on main thread (no sync reads)

  Architecture: dedicated BlobWorker (src/utils/blob-worker.js).
  Main thread communicates via postMessage. JournalMediaStore API
  stays identical — the change is the backing store, not the contract.
  Fallback: if OPFS unavailable or fails write-read-verify test on
  first launch, stay on IDB Blobs (current behavior).

  Gate: profile JournalMediaStore I/O under real usage BEFORE building
  this. If 90th percentile write latency is <200ms (likely for typical
  memo sizes of 50-500 KB), the optimization isn't worth the Worker
  complexity + Safari risk. Build only if profiling justifies it.

  Exit criteria (W7 complete):
    [ ] Schema versioning: every IDB store has a version integer +
        migration chain. Existing migrations (annotations, links)

  W8.2 — CSS @layer architecture (eliminate all !important)
  ─────────────────────────────────────────────────────────
  Problem: 24 !important declarations remain in app.css. Research
  classified them:
    - 11 annotation/highlight display (prevent palette bleed)
    - 6 light-mode palette guards (state-specific overrides)
    - 2 visibility/drag states
    - 5 structural/spacing overrides

  All 24 are load-bearing — they exist because the cascade order of
  existing rules doesn't match the intended priority. Removing them
  without restructuring the cascade would break visual output.

  Fix: CSS @layer (supported in all browsers since 2022, Safari 15.4+,
  Chrome 99+, Firefox 97+). Define explicit cascade layers:

    @layer reset, base, layout, components, annotations, states, overrides;

  Rules in later layers win over earlier layers regardless of
  specificity. This means:

    - `@layer annotations` contains .hl-note, .hl-underline, per-color
      palette rules. No !important needed — the layer order ensures
      annotation rules beat component rules.
    - `@layer states` contains .is-active, :hover, .dragging, .capturing-
      thumb. These win over annotations by layer order.
    - `@layer overrides` is the final layer — currently empty, reserved
      for future escape hatches. If any !important survives the
      migration, it goes here with a comment explaining why.

  Migration approach:

  W8.2a — Layer scaffold:
    Add `@layer` declaration at top of app.css. Wrap existing rules in
    their appropriate layers using `@layer X { ... }` blocks. NO rule
    changes — just wrapping. Verify zero visual change via screenshot
    diff (tools/smoke.js can capture screenshots for comparison).

  W8.2b — Remove !important one-by-one:
    For each of the 24 declarations:
      1. Remove `!important`
      2. Verify the rule still wins (it should, because layer order
         guarantees it)
      3. If it doesn't win, the rule is in the wrong layer — move it
      4. Screenshot-diff to confirm zero visual change
    Same one-commit-per-item discipline.

  W8.2c — Verify dark + light mode:
    Full visual walk in both themes at 3 breakpoints (mobile 375px,
    tablet 768px, desktop 1440px). Every annotation color (10 colors)
    × active/inactive × note/highlight/underline × dark/light = 120
    visual states. Spot-check representative samples (not all 120).

  Risk: MEDIUM — @layer is a cascade-order change that could have
  unexpected interactions with the existing specificity hierarchy.
  Mitigation: the scaffold step (W8.2a) wraps without changing, so
  any breakage from the layer order itself is caught before any
  !important is removed. The removal step (W8.2b) is one-at-a-time
  with screenshot verification.

  Browser support: @layer is in our support matrix (minSdk 26 =
  Android 8 = Chrome 99+ in WebView; desktop Chrome/Edge/Firefox all
  support since early 2022). No polyfill needed.

RISK REGISTER
───────────────────────────────────────────────────────────────────────────────

  HIGH — IndexedDB sync render-path (W2 Tier 3)
    The 5 stores read synchronously in render paths (annotations, notes,
    links, state, prophecy-cards) must be in-memory before React renders.
    The hydration gate is the mitigation, but boot-time latency may
    increase. Measure: time from page load to first render, compare
    pre/post migration. Target: < 200ms overhead.

  HIGH — Safari 7-day data eviction (W2.7 Scenario 3)
    Safari evicts all script-writable storage after 7 days without user
    interaction. Home-screen PWAs are exempt. Users who create data in
    a Safari tab without installing will lose everything. Mitigation:
    intercept first data-creating gesture with a modal explaining the
    risk + install instructions. Can't fix the platform constraint —
    can only warn clearly enough that the user acts before data loss.

  MEDIUM — Export/import media round-trip (W2.6)
    exportPersonalData currently drops all journal media blobs (audio,
    images). Fix adds async base64 encoding of all blobs into the JSON
    payload. For users with >100 MB of media, this may OOM or produce
    a multi-hundred-MB JSON file. Mitigation: 100 MB guard warns user;
    ZIP-based streaming export deferred to W7.

  MEDIUM — iOS PWA storage isolation (W2.4 + W2.7 Scenario 4)
    Installing a PWA on iOS creates a NEW storage container. Data from
    the Safari tab is invisible. Users who build up data in Safari then
    install lose access to it. Mitigation: detect empty-storage-on-
    first-PWA-boot and show an export/import bridge card. Can't fix
    the platform constraint — Apple isolates by design.

  MEDIUM — Budget Android near-zero quota (W2.7 Scenario 7)
    Chrome/WebView quota = 60% of FREE disk. Devices with <500 MB free
    get quotas as low as a few MB. App must detect and warn, not crash.
    Mitigation: startup health check + QuotaExceededError handling on
    every write path → graceful degradation to read-only mode.

  MEDIUM — Audio recording cross-browser (W1.4)
    MediaRecorder API: Chrome supports audio/webm, Firefox supports
    audio/ogg. Safari doesn't support MediaRecorder reliably. Need
    to test mime negotiation and base64 encoding on all three browsers.
    Mitigation: mime detection at start time, fallback to audio/wav
    if needed.

  MEDIUM — Service worker cache size (~12.5 MB total)
    Full corpus cache is large. Some browsers may throttle or reject.
    Mitigation: cache critical path on install (~2.3 MB), lazy bundles
    on first fetch. Monitor navigator.storage.estimate() for pressure.

  MEDIUM — hlTick removal blast radius (W7.3)
    39 files touched. The 8 truly-live uses must be migrated to
    useSyncExternalStore before the plumbing can be removed. If any
    live use is missed, annotation re-renders will silently stop
    working. Mitigation: smoke walk every annotation operation (add,
    remove, color change, note create, bookmark toggle, link add)
    after W7.3a before starting W7.3b.

  MEDIUM — CSS @layer cascade rewrite (W8.2)
    Moving 4,100 lines of CSS into @layer blocks changes the cascade
    globally. Unexpected interactions could break any visual state.
    Mitigation: scaffold step (W8.2a) wraps without removing any
    !important; breakage from layer order itself is caught before
    any rule changes. Screenshot comparison at each step.

  LOW — CSP compatibility on GitHub Pages
    'self' should resolve correctly. If not, fall back to explicit
    origin in CSP. Easy to diagnose via console errors.

  LOW — Font licensing
    Cinzel + EB Garamond are OFL (SIL Open Font License). Bundling
    in an APK and serving via GitHub Pages are both permitted uses.
    No license risk.

  LOW — UI typing flood (W8.1)
    82 files × ~5 type errors each = ~400 initial errors. But Q4
    proved the pattern: file-by-file, 4-7 files/day, zero-tolerance
    after each. Risk is tedium, not breakage.
