3.2 DATA FORMAT FINDINGS  (from Explore agent + cross-checked with CLAUDE.md)

  EXECUTIVE SUMMARY:
    ~28 data files, ~46 MB total. Two primary format families plus Bible/Studies.
    The split between Format A (Volumes, blocks/segments) and Format B (WTLB,
    paragraphs with inline {{ref:...}} markup) is the central data-layer pain
    point. Holy Days bridges both with a `type` discriminator.

  PRIMARY FORMATS:
    Format A (letters/studies): blocks: [{type, segments|lines|text}]
    Format B (WTLB/Blessed): paragraphs: [{align, text-with-inline-markup}]
    Format C (Bible): {bookId: {chapters: [{sections: [{verses: [{n, text}]}]}]}}
    Format D (Matthew Study): {chapters: [{verses: [{n, text-with-embedded-segments}]}]}

  COLLECTIONS BY FORMAT:
    Format A:
      volume-two.js (LETTERS) — gold standard, UNQUOTED keys
      volume-{one,three,four,five,six,seven}.js — JSON-quoted keys
      lords-rebuke.js, letters-flock.js, letters-timothy.js
      hidden-manna.js
      bible-studies.js (with Format A-style blocks per chapter)
    Format B:
      wtlb-one.js, wtlb-two.js (WTLB_ONE, WTLB_TWO arrays)
      the-blessed.js (THE_BLESSED + THE_BLESSED_SCRIPTURES)
      wtlb-scriptures.js (shared NKJV dict)
    Mixed (A + B):
      holy-days.js (entry.type discriminates: 'wtlb' uses paragraphs[],
                    'letter' uses blocks[])
    Bible:
      books.js (NKJV master), books-restored.js (restored names overlay)
      matthew.js (Matthew Study Bible — Format D-ish)
      matthew-nkjv.js (flat ref→text dict for Matthew Study scriptures)
      matthew-plain.js (plain NKJV Matthew, Scriptures corpus only)

  BLOCK TYPES (Format A):
    para            — paragraph with segments[]
    poetry          — verse with lines: Segment[][] OR segments: Segment[]
    closing         — letter sign-off, text: string
    closing-fn      — closing with attached footnote, segments: Segment[]
    intro           — introduction text (preface), segments: Segment[]
    heading         — h2/h3 section heading, level + text
    note            — editorial note, text: string
    scripture       — quoted scripture block (rare)
    prophecy-group  — Matthew Study collapsible cards, segments + cardType
    cover-image     — study cover, src
    study-image     — study diagram, src + alt + caption

  SEGMENT TYPES (inline `t`):
    text            — plain text, v: string
    italic          — _italic_, v: string
    bold-italic     — _**bold-italic**_, v: string
    caps            — ALL CAPS, v: string
    fn              — footnote ref, v: "1" (links to footnotes['1'])
    stanza-break    — empty line in poetry
    letter-link     — cross-collection ref, label + link: {collection, letterTitle}

  INLINE MARKUP (Format B):
    _italic_                              underscore wrap
    **bold**                              double asterisk
    _**bold-italic**_                     combined
    {{ref:Book Chapter:Verse}}            scripture popup
    {{ref:A:B; C:D}}                      compound (semicolon-separated)
    {{nav:bookId:chapter}}                Bible chapter nav
    †                                     section divider character
    ~ "From X" attribution                tilde-prefixed
    [N], [[N]]                            legacy footnote markers (mostly cleaned)

  FOOTNOTE TYPES (Format A):
    { type: "scripture", ref: "Isaiah 13:11" }
    { type: "scripture", ref: "...", seeAlso: { collection, letterTitle, label, excerpt } }
    { type: "note", text: "Also read X", link: { collection, letterTitle } }
    { type: "note", text: "X", url: "https://..." }

  CROSS-REFERENCE PATHS:
    prevLetter/nextLetter         (Format A sequential)
    prevEntry/nextEntry           (Format B sequential — Holy Days uses these)
    metaAddendum + metaAddendumLink + metaAddendumInternal + metaAddendumUrl
    relatedTopics: [{ label, url | link | internalStudy }]
    bibleStudies: [{ label, url | link | internalStudy }]
    addendum (Also Read card)
    Letter-link segments inside study text
    Holy Days sourceLabel field (origin annotation)

  MEDIA URL FIELDS:
    audioUrl       (Bandcamp track per letter — most letters)
    soundcloudUrl  (subset: V1, V3, Timothy)
    videoVoiceUrl  (YouTube voice-over)
    videoVoiceLabel(optional label)
    videoMusicUrl  (YouTube music-set)
    videos: [{label, url}]   (legacy array form, rare)

  KNOWN BUG TAXONOMY (D1–D10 from CLAUDE.md §6.6):
    D1  — ref string contains the entire verse text crammed in
    D2  — multi-verse markers fail to render gold sup
    D3  — defunct [N] body markers (orphaned bracket numerals)
    D4  — translation-tagged ref missing nkjv entry
    D5  — doubled verse markers (Unicode ⁰¹²³ + decimal)
    D6  — doubled superscript without text duplication
    D7  — cite parser glitch (stray prefix)
    D8  — glued text near refs (no space between word and {{ref:)
    D9  — compound refs not split with " | " separator
    D10 — mixed footnote affordances on same content
    Per CLAUDE.md §14: 54 D-pattern bugs were fixed in prior pass; spot-check
    needed for any new ones introduced or missed.

  FORMAT INCONSISTENCY MATRIX (most painful for Phase 2):
    1. blocks vs paragraphs (incompatible content shape)
    2. segments[] with t-types vs inline {{ref:}} string markup
    3. nkjv vs scriptures vs WTLB_SCRIPTURES (3 different dict locations)
    4. prevLetter vs prevEntry naming
    5. UNQUOTED keys (V2) vs QUOTED keys (everyone else)
    6. Footnote types (scripture/note) vs no footnotes (B uses inline only)
    7. Holy Days needs `type` discriminator at runtime

  PROPOSED UNIFIED SCHEMA (from agent + my refinement):
    Single `Entry` shape with:
      - id, num, title, date?, from?, spoken?, forLine?
      - content: Block[]                  (uniform — no Format A/B split)
      - footnotes?: { [key]: Footnote }   (numbered or scripture-tagged)
      - scriptures?: { [ref]: string }    (per-entry NKJV/alt overrides)
      - media?: { audio?, video?, transcript? }
      - nav?: { prev?, next?, related?[], addendum? }
      - tags?: string[]                   (for Holy Days topical, search)
      - source?: { sourceLabel? }         (for Holy Days re-attributions)
    Block.type ∈ {paragraph, poetry, heading, image, divider, closing, prophecy-group, ...}
    Block.content: Segment[] (uniform)
    Segment.type ∈ {text, emphasis, strong, footnote, link, scripture-ref, divider, line-break}

  MIGRATION STRATEGY (proposed by agent — I'll refine in Phase 2):
    A. Build normalize(rawEntry) → UnifiedEntry function (NO data file changes)
    B. Audit current data with normalizer + bug detector
    C. Fix identified bugs in source data
    D. Generate unified data files in parallel with old (dual-format support)
    E. Update renderers to consume only unified format
    F. Retire old format files

SECTION 4 — IMPROVEMENT STRATEGY (high-level shape)
───────────────────────────────────────────────────────────────────────────────

THE CHALLENGE IN ONE SENTENCE:
  This is a 12,482-line single-file React app that works well, has rich
  features, professional design, and deep content — but has NO build system,
  NO tests, NO type safety, TWO incompatible data formats, accessibility
  gaps, and a 2,350-line App() function that has reached the practical
  limit of single-file maintainability.

THE GOAL IN ONE SENTENCE:
  Refactor the codebase into a modular, maintainable, testable foundation
  that makes the planned features (Notebooks, Bookmarks, Journal, AI Space,
  TTS, in-app video, cloud sync) implementable without compounding the
  existing complexity — while NEVER breaking a single working feature.

GUIDING PRINCIPLES:
  1. NEVER break a working feature mid-refactor. Verify in browser after
     every batch of changes. The user's data (highlights, notes, links,
     history, settings) MUST migrate forward losslessly.
  2. Refactor INCREMENTALLY. Each phase ships value on its own. Stopping
     at the end of any phase leaves the codebase BETTER than before.
  3. Data first, code second, build last. Clean data lets us simplify
     renderers. Modular renderers let us add features cleanly. Build
     tools come once we know what code we have.
  4. Single-file index.html may stay forever. The MODULAR source files
     can simply be re-concatenated at build time if we want one HTML file
     for the Android WebView. (The Android shell expects index.html in
     the assets folder; we keep that shape.)
  5. Honor the content. The annotation system, scripture refs, cross-refs,
     and reading-chain navigation are MORE than features — they are the
     core respect for the source material. Refactoring must preserve
     behavior exactly.
  6. Coordinate with parallel work. The Notebooks feature is being built
     in another session. We do not edit the annotation/note storage layer
     in ways that conflict. We extract WHAT exists; we don't redesign it
     while it's being extended.

THE SHAPE OF THE PLAN (5 phases, each ~1-3 weeks):

  PHASE 1 — Bug fixes & quick wins (1-3 days)
            Pure additive. Zero risk. Ship immediately.
            Accessibility (focus indicators, prefers-reduced-motion).

            scripture-text resolution unification (NKJV/scriptures/
            wtlb-scriptures all become one lookup chain).

  PHASE 3 — Code modularization (2-3 weeks)
            Extract index.html into ~40 modules. Pure logic first
            (stores, COLLECTIONS, scripture-refs, nav-index). Then UI

2.1 DESIGN THE UNIFIED SCHEMA (1-2 days)

  [ ] Write the schema in JSDoc comments + a typescript-style declaration
      file (data-schema.d.ts) at project root. NOT enforced at runtime
      yet, but documented:

      /**
       * UnifiedEntry — single shape for letters, WTLB entries,
       * Blessed sections, Holy Days entries, Bible Studies chapters,
       * and Hidden Manna.
       */
      interface UnifiedEntry {
        id: string;
        num: number;
        title: string;
        date?: string;            // letters only
        from?: string;            // letters only
        spoken?: string;          // letters only
        forLine?: string;         // letters only
        preamble?: string;        // optional
        sourceLabel?: string;     // Holy Days re-attribution
        content: Block[];         // unified, no format split
        footnotes: Record<string, Footnote>;
        scriptures: Record<string, string>;  // local NKJV/alt overrides
        media?: { audio?: AudioLink[], video?: VideoLink[] };
        nav?: { prev?: EntryRef, next?: EntryRef,
                related?: RelatedLink[], addendum?: AlsoRead };
        tags?: string[];
        kind?: 'letter' | 'wtlb' | 'blessed' | 'holy-day' | 'study' | 'hidden';
      }

      interface Block {
        type: 'paragraph' | 'poetry' | 'heading' | 'image'
            | 'closing' | 'closing-fn' | 'divider' | 'prophecy-group'
            | 'cover-image' | 'study-image' | 'note' | 'scripture';
        id?: string;
        align?: 'left' | 'center' | 'justify';
        level?: number;           // for headings
        src?: string;             // for images
        text?: string;            // for closing/note/divider
        content: Segment[];       // for everything else (uniform!)
      }

      interface Segment {
        type: 'text' | 'emphasis' | 'strong' | 'caps'
            | 'footnote-ref' | 'letter-link' | 'scripture-ref'
            | 'nav-ref' | 'stanza-break' | 'line-break';
        text?: string;
        // For footnote-ref:
        footnoteId?: string;
        // For letter-link:
        link?: { collection: string, letterTitle: string, excerpt?: string };
        // For scripture-ref:
        ref?: string;             // "Matthew 4:4"
        translation?: string;     // 'NKJV' (default), 'ASV', 'KJV', etc.
        // For nav-ref:
        bookId?: string; chapter?: number;
      }

      interface Footnote {
        id: string;
        type: 'scripture' | 'note';
        text?: string;            // for note
        scripture?: { ref: string, translation?: string };
        link?: { type: 'internal' | 'external', target: any };
        seeAlso?: { collection, letterTitle, label?, excerpt? };
      }

PHASE 4 — BUILD SYSTEM + TOOLING (esbuild + JSDoc + lint + tests)
Estimated effort: 1-2 weeks. Risk: low (additive — keeps old build working).

GOAL: Replace cat-modules.py with a real bundler. Add type annotations
      via JSDoc (no TypeScript compile step). Add ESLint with a
      permissive ruleset. Add a few critical unit tests.

WHY THIS ORDER: Phase 3 gave us modules. Phase 4 gives us safety nets and
                better DX. Without Phase 3 first, Phase 4 has nothing to
                bundle/lint/test.

WHY ESBUILD: It's a single binary. No config required. Builds in <1 second.
             Has built-in JSX support (lets us optionally rewrite the
             pre-compiled React.createElement to JSX without committing
             to TypeScript). No node_modules tax — keeps the project lean.

WHY JSDoc INSTEAD OF TYPESCRIPT: TypeScript adds significant build complexity
                                  and a learning curve. JSDoc gives us type
                                  hints in editor + tsc-based typechecking
                                  (without the build step) — about 80% of
                                  the safety. We can migrate to .ts files
                                  later if we want.

4.1 INSTALL ESBUILD

  [ ] npm init -y (creates package.json)
  [ ] npm install --save-dev esbuild
  [ ] Write build.mjs (esbuild config):
      - entryPoints: ['app/src/main/assets/src/boot.js']
      - bundle: true
      - format: 'iife'
      - outfile: 'app/src/main/assets/index-bundled.js'

SECTION 10 — TRAPS / GOTCHAS LEARNED (append as discovered)
───────────────────────────────────────────────────────────────────────────────

10.1 The CSS lives inside `const CSS = \`...\`` template literal.
     Any backtick or `${` inside the CSS string terminates or interpolates
     the template, silently breaking the CSS const to evaluate to the
     global window.CSS namespace. React then crashes with "Minified React
     error #31". (CLAUDE.md §17.11b, end.)

10.2 Volume Two uses unquoted JS keys (`id: "..."`); all other volumes
     use JSON-quoted keys (`"id": "..."`). Both work; don't "normalize"
     during routine edits — wait for the Phase 2 migration script.

10.3 The C: junction can confuse: `C:\...\VOTReader-studio\app` is a
     Junction → `D:\VOTReader-studio\app`. Always edit on D:.

10.4 The preview tool starts http.server from CWD (not from a configured
     directory). The C:\ junction makes this work transparently because
     the preview serves D:\ files via the junction. If preview ever stops
     reflecting changes, check the junction with:
     `cmd /c dir C:\Users\corbi\OneDrive\Desktop\VOTReader-studio` (look
     for `<JUNCTION>` flag on `app`).

10.5 Pre-compiled React.createElement is verbose and hard to spot typos
     in. When extracting components in Phase 3, consider opportunistic
     JSX conversion (esbuild handles the transpile) — but do NOT do a
     mass rewrite. One file at a time, verify, move on.

10.6 The Notebooks feature is being built in another session. Any change
     to AnnotationStore or NoteStore risks merge conflict. If you have to
     touch them, message-coordinate first or work in a separate branch.

10.7 ~46 MB of data files is a LOT to ship in a WebView. The 7 alt-
     translation Bible files (~4.5 MB each = ~32 MB) are lazy-loaded —
     don't accidentally hoist them into eager loads.

10.8 The user is solo-maintaining. Tooling complexity must justify itself.
     Phase 4's bundler/linter/tests are all leans — don't bloat into
     enterprise-grade CI overengineering.

10.9 (FROM 2026-05-10 NOTES SESSION) CSS template literal escapes.
     Stray backticks in CSS comments terminate `const CSS = \`...\``
     mid-string. Stray `\<digit>` sequences (like `\2060`) are
     interpreted as illegal octal escapes. Both cause `CSS` to resolve
     to `window.CSS`, which React then can't render → Minified React
     error #31 → black screen. The Notes session tripped over both
     at least twice. Search the CSS template region for stray
     backticks BEFORE assuming the bug is elsewhere.

10.10 (FROM 2026-05-10 NOTES SESSION) tabField setters call their
      updater functions ASYNC. Reading state via
      setX((prev) => { sideEffect; return ...; }) does NOT work for
      synchronous reads — the side effect runs in a future tick.
      Use the matching ref instead.

SECTION 11 — CANONICAL UX PATTERNS (established 2026-05-10, MUST follow)
───────────────────────────────────────────────────────────────────────────────

The Notes/Notebooks overhaul (parallel session, completed 2026-05-10)
established these app-wide patterns. Phase 5 features (Bookmarks, Journal,
Highlights browser, AI, TTS, Video) MUST follow them — diverging would
fragment the UX.

11.1 NO INSTANT DELETE — EVER.
     Every destructive action follows: tap → menu → inline confirm strip.
     Examples: AnnotationActionChip's Remove button morphs into "Remove
     this highlight? [Cancel] [Yes, remove]" inline. Notebook delete:
     "Notes will move to Uncategorized [Cancel] [Yes, delete]".
     Future Bookmarks/Journal deletes follow the same shape.

11.2 SHEET NESTING (LinkPicker pattern).
     Bottom sheets render as: <overlay class="...overlay"><sheet>...</sheet></overlay>
     The overlay is the dim+backdrop; the sheet is the inner panel.
     The sheet has e.stopPropagation() on click. Result: tapping the
     dim closes the sheet; tapping inside the sheet doesn't.
     Nesting prevents the "everything looks washed out" stacking bug.

11.3 SINGLE-SHOT BACK PILL (tapThroughBack + destSnapshot).
     The OLD multi-level fromLetterStack pattern still exists for letter↔
     letter tap-through. The NEW pattern for cross-screen-context tap-through
     (e.g., Notes index → source chapter → Back to My Notes) uses:
       - destSnapshot = expected destination state captured at push time
       - The pill renders only while current state matches destSnapshot
       - Navigating to a DIFFERENT chapter than expected → pill hides + stack prunes
     Phase 5: "Back to My Bookmarks", "Back to My Highlights", "Back to My Journal"
     all use this same helper. Do NOT add another back-stack mechanism.

11.4 WORD-BOUNDARY SNAPPING (snapRangeToWords).
     Any code that creates a user-anchored object on text (highlight,
     underline, note, bookmark) MUST run the user's selection through
     snapRangeToWords() first. Convention: same as Kindle/Apple Books/
     LDS Gospel Library — selections always expand outward to whole words.
     Mid-word boundaries cause icon-positioning bugs ("kicks the highlight
     to the next line").

11.5 MULTI-MEMBERSHIP TAGGING (no colors on the bucket).
     Notebooks established the model: name only, no color/icon, an entry
     can be in 0, 1, or many buckets. Toggling membership is instant (no
     "Done" button). Future tagging systems (Bookmark tags, Journal tags,
     AI saved-searches) follow this — bucket has identity but no visual,
     the visual lives on the entry.

11.6 LIBRARY IS THE HOME FOR PERSONAL STUDY.
     The 2x2 Library grid (Notes / Journal / Bookmarks / Highlights &
     Underlines) is the canonical home for everything user-generated.
     Do NOT add personal-study features as new top-level home tiles —
     fill the existing Library tiles. Library subtitle: "Personal Study".

11.7 ICONS FOLLOW ENTRY COLOR.
     The 📝 note icon's SVG stroke is set per-color from the note's
     hl-{color} class. Bookmark/journal icons (when added) should follow
     the same convention so users see-at-a-glance the entry's category.

11.8 OVERLAP IS A FEATURE, NOT A BUG.
     Multiple notes/highlights can stack on the same passage. The sweep-
     line rendering in HighlightableText handles this. When tap targets
     overlap (icons or marks), a disambiguation popover lists each option.
     New annotation-like features (bookmarks?) should respect this — don't
     enforce uniqueness at create time.

11.9 SAVE DISABLED ON EMPTY/WHITESPACE.
     NoteSheet's Save button is disabled until body.trim() is non-empty.
     Apply this principle anywhere user content is required (Journal body,
     Bookmark label if required, etc.). Empty saves should be impossible.

11.10 SHEETS AUTO-CLOSE ON SAVE.
      NoteSheet edit → Save → sheet closes (no "back to read first" step).
      The user's intent on Save is "I'm done." Don't force an extra tap.

17.1 I PROPAGATED AN UNVERIFIED CLAIM FROM THE PARALLEL SESSION'S WORK
     The "snapRangeToWords regex range bug" — claim that `[\w'‘-]`
     is parsed as a range including `()*+,` — is WRONG. Hyphen at

     with me, I integrated it into PLAN.txt and elevated this claim
     to P0 (Phase 0.15) without re-verifying it against the actual

     they later created as improvement2.txt) caught it.

     What I actually did this session:
       - Dispatched 6 round-1 agents + 4 round-2 agents (10 total)
       - VERIFIED some claims via direct code reads (destSnapshot bug
         confirmed by tracing setLetterId calls; applyNoteIcons

       - Did NOT verify when ingesting improvment.txt's findings —
         the bogus regex claim got promoted to P0 here on trust alone

     LESSON: Treat ANY unverified claim as a hypothesis, whether it
     came from my own agents or from another session's work shared
     into my context. The verification cost is low; the cost of being
     wrong (and ranking it P0) is high. When ingesting another
     session's output, the same "verify before propagating" rule
     applies as it does for my own agents.

17.2 I NEVER ASKED THE USER WHAT THEY ACTUALLY FIND BROKEN
     12 audit agents producing 100+ findings, but the highest-signal
     information was a 3-sentence answer the user could have given me
     in 30 seconds: "The app feels underfinished, footnotes are buggy,
     UI is paper-cut city." That changed the entire priority ordering.

     LESSON: For any "improve this" task, ask the user FIRST what bothers
     THEM. Don't audit blindly. The user's pain is the priority signal;
     everything else is hypothesis-generation.

17.3 I OPTIMIZED FOR FINDING BUGS, NOT FOR IMPACT
     "Improve the app" became "list every issue." The result was a
     3,000-line audit when a 50-line action plan would have been more
     useful. Quantity ≠ quality. I never produced a "if you only fix
     3 things, fix these" version.

     LESSON: For working memory documents, length is a cost, not a
     feature. Aim for "one screen of action items, ranked." Push the
     details into appendices.

17.4 I NEVER ZOOMED OUT ON PRODUCT
     Zero competitive analysis. I never asked: what do reading apps
     standardly have that this one lacks? (Reading plans, audio sync,
     daily prompts, family share, sepia/cream paper modes, font/size
     controls beyond on/off, etc.) I audited what IS there; never asked
     what's MISSING that matters to a user.

     LESSON: For product work, do a competitive scan early. YouVersion,
     Logos, Olive Tree, the Lectionary apps — what do they ship as
     standard that this one doesn't? Frame audit findings against the
     baseline of "what reading apps do."

17.5 I NEVER READ THE ACTUAL CONTENT
     This is a Bible/letters reader. The reading experience IS the
     product. I had agents look at file sizes, schemas, CSS contrast
     ratios. Not one of us actually OPENED a letter and asked "is
     reading 'The Wide Path' in this app a pleasant experience? Is
     the typography breathing right? Is line length comfortable? Does
     the rhythm of paragraphs feel right?"

     For a reading app, that's the entire UX question. We answered every
     OTHER question.

     LESSON: For a content app, evaluate the content experience first.
     Code quality is moot if reading the content sucks.

17.6 PWA WAS COMPLETELY ABSENT FROM MY PLAN
     The app is a WebView wrapper around HTML/JS that doesn't actually
