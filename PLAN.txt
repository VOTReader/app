POST-Q3 / POST-Q4 BACKLOG (priority order)
───────────────────────────────────────────────────────────────────────────────

1. SMOKE-LITE.JS — Node-runnable subset of tools/smoke.js for CI
   ─────────────────────────────────────────────────────────────
   tools/smoke.js (34 KB, ~681 lines) is browser-only — runs via preview_eval,
   only when triggered. CI can't use it, so regressions slip until manual smoke.

   Extract the non-DOM subset into Node:
     - Every .js in app/src/main/assets/src parses cleanly (overlaps CI syntax-check)
     - All COLLECTIONS entries resolve to non-empty letter arrays or a preface
     - Module export validation — lexical-mirror writes 143 globals; verify
       source-of-truth list matches what entry files export
     - No circular imports in stores/_entry-b.js, renderer/_entry.js,
       ui/_entry-d.js (esbuild rejects strict cycles, but reorder regressions
       inside an entry file are not always caught)

   Effort: ~half-day. Higher regression-coverage value than the entire Q3.3e
   warning sweep combined.

   Schedule: BEFORE Q5 (Vitest). smoke-lite is the safety net Q5's test
   architecture rides on top of.

2. useNavHistoryTracking EXTRACTION (app.jsx:814)
   ──────────────────────────────────────────────
   The auto-track history effect that adds entries to readHistory on screen
   change. use-history.js header lines 13-14 explicitly documents this as
   a future extraction candidate that stayed in App() because of three
   App()-local helper deps (_findLetter, getStudyById, addToHistory).

   Signature: `useNavHistoryTracking({navState}, {addToHistory, _findLetter,
   getStudyById})`. Drops Q3.3e-app-jsx's most architecturally-debted disable
   cite (the cite at app.jsx:816 explicitly names this as the candidate).

   Schedule: P6m, post-Q4. App() shrinks 30-50 more lines.

3. useSyncExternalStore MIGRATION (eliminates Bin 4 hlTick cites)
   ─────────────────────────────────────────────────────────────
   Q3.3e added 24 disable cites for the canonical hlTick cache-bust pattern:

       const x = React.useMemo(() => SomeStore.get(hlKey), [hlKey, hlTick]);

   hlTick is a bump-counter that increments on store mutations, signaling
   memos to re-read. ESLint flags it "unnecessary" because it can't see
   through the store-read boundary. Removing it breaks highlight/bookmark/
   note refresh.

   React 18's useSyncExternalStore replaces this idiomatically:

       const x = React.useSyncExternalStore(store.subscribe, () => store.get(hlKey));

   Each store grows `subscribe(cb) → unsubscribe` + `getSnapshot()`. Consumers
   replace tick-counter useMemo with useSyncExternalStore. No false-positive
   lint. Concurrent-renderer-safe.

   Scope: refactor — 6 stores + 16 consumer sites. Changes runtime re-render
   semantics (consumers re-render more precisely, not on any mutation).
   NEEDS Q5 tests in place first.

   Schedule: Q5 or Q6.

4. BUNDLE-A.JS LAZY-LOAD (UX, post-Q6)
   ─────────────────────────────────────
   bundle-a.js is 11.7 MB loaded on every cold WebView boot. On mid-range
   Android, ~3-5 seconds of parse time before any UI paints.

   Split into:
     - bundle-a-core.js (~2 MB): vendor + search engine + small registries
     - bundle-a-corpus-{volKey}.js (lazy): each Volume/WTLB/etc.

   Load on-demand when user opens a collection. The single biggest UX win
   available. Significant refactor — needs COLLECTIONS registry to gate
   corpus loading + a fetch-and-await wrapper that doesn't break the
   synchronous `colLetterArr` contract.

   Schedule: post-Q6, deliberately scheduled (not a polish item — a real
   architectural change with smoke regression risk).

5. APP() DECOMPOSITION (between Q5 and Q6, OR interleaved with Q5)
   ────────────────────────────────────────────────────────────────
   App() = 2,191 lines after P6's 15-hook extraction. Cannot meaningfully
   type (Q4 scope deliberately excludes) or test (Q5 scope unclear without
   decomp). Likely needs another wave of extraction:
     - Render-tree segments → component files
     - Remaining state-coordination glue → 1-2 new hooks
     - Inline nav-helpers (goHome/goNavOrigin/goSearchOrigin/etc.) → useNav?
   Target: get App() under 800 lines. Each extraction follows the P6 workflow
   (smoke gate, OWNS/PARAMS/RETURNS header, line-count tracking).

   Schedule: after Q5 tests exist (need regression coverage for the
   decomposition); could interleave with Q5 if smoke-lite + Vitest land
   incrementally.

INFRA / TOOLING ITEMS (one-liners — open small, ship fast)
───────────────────────────────────────────────────────────────────────────────

- lint:globals AUTO-REGEN in pre-commit hook: today the hook rebuilds bundles
  but does NOT run `npm run lint:globals`. If a dev adds a new global export,
  the committed globals file lags reality until CI regenerates it. Mirror
  the bundle-rebuild logic: regenerate + restage globals on any source change.

- LINT-STAGED auto-fix mode: currently `eslint --max-warnings 0` (no --fix).
  If devs want auto-fix on `--fix`-able warnings (unused-vars `_` prefix,
  whitespace), add a second config entry. Low priority — auto-fix can hide
  intentional patterns.

- `tsc --noEmit` IN CI: when Q4 lands, add as a CI step parallel to lint.
  Same `--max-warnings 0`-style gate (typecheck failures fail CI).
