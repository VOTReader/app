   Phase 2 sequencing — low-risk PILOT first, high-risk after:

     P7-2a (FIRST):  ScreenRouter pattern over the 26 trivial 4-line
                     letter/index wrappers. ~104 lines collapsed by a
                     lookup table replacing the switch chain. Near-zero
                     risk: the wrapper div stays identical (each entry
                     was already `{screen === X && <XScreen … />}` — a
                     dispatch table renders the same XScreen with the
                     same props, just chosen by `screens[screen]`). No

                     effect-fire semantics change. The mechanical win
                     proves the ScreenRouter shape works before any
                     substantive screen extraction touches it.

     P7-2b, 2c, …:   Substantive screen extractions in size-descending
                     order: garden-view (240) → bible-study-chapter (87)
                     → matthew-ch (42) → bible-ch (40) → holy-days-index
                     (37). Each is its own commit. The risks below
                     (CSS, useEffect, ErrorBoundary) apply ONLY to
                     these — but they ride on the ScreenRouter pattern
                     proven in 2a.

   Workflow per substantive extraction (P7-2b onward, NOT 2a):
     1. Extract the screen block into ui/screens/XScreen.jsx (most
        screens already have this file shape; this is moving the
        last conditional-render wrapper into the screen file itself).
     2. Update the App() conditional to a single `<XScreen … />` call
        (or, eventually, collapse into ScreenRouter).
     3. Run smoke walk (votSmoke or in-browser nav of the affected
        screen) — NOT just vitest. Render extraction can break CSS
        selectors that target `.app > .screen-container > .X` even
        when component logic is preserved.
     4. Visual diff: before/after screenshot of the screen. Layout
        shifts from a changed component hierarchy are silent to tests.
     5. ErrorBoundary placement: if a screen had an inline error
        wrapper, the extraction must preserve it (or the boundary
        moves with the component — verify on purpose-broken render).

   Render-tree risks (call out so they don't surprise):
     - CSS selectors that traverse the App-level wrapper (`.app > .x`)
       break if the component hierarchy changes depth.
     - `useEffect(..., [screen])` inside child components fires
       differently from the App-level one (the parent's screen change
       unmounts the child rather than re-running its effects).
     - Sheet/portal placement: screens that render sheets via portals
       still work after extraction, but sheets that READ closure values
       from App() need explicit props.

   Realistic Phase 2 reduction: ~700–800 lines (5–7 biggest screens
   extracted + ScreenRouter for the 26 trivial dispatchers + ~100 lines
   of boilerplate eliminated). Lands App() at ~700–900.

   <800 IS achievable, but it requires BOTH phases. Do not promise
   <800 from Phase 1 alone.

2. useSyncExternalStore MIGRATION (eliminates Bin 4 hlTick cites)
   ─────────────────────────────────────────────────────────────
   Q3.3e added 24 disable cites for the canonical hlTick cache-bust pattern:

       const x = React.useMemo(() => SomeStore.get(hlKey), [hlKey, hlTick]);

   hlTick is a bump-counter that increments on store mutations, signaling
   memos to re-read. ESLint flags it "unnecessary" because it can't see
   through the store-read boundary. Removing it breaks highlight/bookmark/
   note refresh.

   React 18's useSyncExternalStore replaces this idiomatically:

       const x = React.useSyncExternalStore(store.subscribe, () => store.get(hlKey));

   Each store grows `subscribe(cb) → unsubscribe` + `getSnapshot()`. Consumers
   replace tick-counter useMemo with useSyncExternalStore. No false-positive
   lint. Concurrent-renderer-safe.

   Scope: refactor — 6 stores + 16 consumer sites. Changes runtime re-render
   semantics (consumers re-render more precisely, not on any mutation).
   Q5 tests are now in place (per [[refactor-after-tests]]) — unblocked.

   Schedule: post-P7 (App() decomposition first), or alongside it if a
   P7 hook extraction naturally touches one of the 16 consumer sites.

3. BUNDLE-A.JS LAZY-LOAD (UX, post-Q6)
   ─────────────────────────────────────
   bundle-a.js is 11.7 MB loaded on every cold WebView boot. On mid-range
   Android, ~3-5 seconds of parse time before any UI paints.

   Split into:
     - bundle-a-core.js (~2 MB): vendor + search engine + small registries
     - bundle-a-corpus-{volKey}.js (lazy): each Volume/WTLB/etc.

   Load on-demand when user opens a collection. The single biggest UX win
   available. Significant refactor — needs COLLECTIONS registry to gate
   corpus loading + a fetch-and-await wrapper that doesn't break the
   synchronous `colLetterArr` contract.

   Schedule: post-Q6, deliberately scheduled (not a polish item — a real
   architectural change with smoke regression risk).
