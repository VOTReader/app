PWA MIGRATION — CROSS-PLATFORM (Android + PC)
═══════════════════════════════════════════════════════════════════════════════

Goal: VOTReader runs as a native Android APK (existing) AND as an installable
PWA on desktop (Chrome/Edge/Firefox). One JS codebase, two shells. All data
stays local on both platforms. No server, no credentials, no API keys.

Rationale — the inversion:

  The architectural review surfaced 18 critiques. The deepest one — "WebView
  fighting native UX" — is the meta-critique behind most of the others.
  Going cross-platform doesn't just ADD a platform. It INVERTS that critique.
  The 135+ modules of battle-tested web code stop being a limitation locked
  inside a single-platform Android shell and become the core asset. PWA
  validates the foundation rather than rebuilding it.

  Concretely, every strength W0-W8 leans on is already in place:
    - AndroidBridge call sites are already guarded with `if (window.AndroidBridge)`
      — desktop browser mostly works today (W1 is formalization, not invention).
    - N1's native extraction (BridgeHost / AppInterface / JsBridge /
      MainViewModel / NativeAudioRecorder / StorageManager) already demonstrates
      the shell-vs-app separation that PlatformBridge will mirror on the JS side.
    - The 6-step pre-commit gate + 476 vitest + 134 JUnit + JaCoCo 0.85 floor
      means changes are inspectable per-commit. The safety net is built;
      this work uses it rather than rebuilding it.
    - 1.03 MB critical-path bundle is small enough to cache as the PWA
      install-time shell (W3).

  One codebase, two shells, all data local. No server, no credentials, no API
  keys, no runtime network egress. The PWA isn't a port — it's revealing what
  was already there.

  W0-W6 makes the app work everywhere. W7-W8 closes every remaining critique.
  When W8 closes, the 18-item traceability matrix at the bottom of this file
  is clean — zero gaps.

  Mandate: not just executing the plan. Stay alert to anything broken or
  missable as we touch each file. Improve as we go. Creative & professional.
  Every commit ships to production.

Current state (pre-migration):
  - The JS app ALREADY mostly works in a desktop browser (React, stores,
    corpus, rendering all load). AndroidBridge calls silently no-op because
    every call site is already guarded: `if (window.AndroidBridge) {...}`.
  - What DOESN'T work on desktop: file import/export, audio recording,
    native screenshots, back-button navigation, fullscreen toggle, zoom,
    haptics, status bar, keep-screen-on.
  - Font files are declared in CSS @font-face but DO NOT EXIST in the repo
    (fonts/ directory is empty). This is a pre-existing visual bug on any
    platform — system fallback fonts render instead of Cinzel/Garamond.
  - localStorage persistence works on both but has quota ceiling (~5-10 MB)
    and no transactional writes.
  - No PWA infrastructure exists: no manifest.json, no service worker, no
    offline fallback, no PWA meta tags.

Sequencing rationale:
  W0 → W1 → W2 → W3 → W4 → W5 → W6

  W0 (prerequisites) unblocks everything: fonts + owed device verification.
  W1 (platform bridge) makes desktop functionally correct — file I/O, audio,
    back-button, screenshots all work via browser APIs.
  W2 (IndexedDB) fixes the persistence ceiling on BOTH platforms — the one
    change with highest user-facing impact.
  W3 (PWA shell) makes it installable + offline on desktop.
  W4 (desktop polish) optimizes the desktop experience after it works.
  W5 (hosting + CI) ships it.
  W6 (cross-platform verification) proves it.

  Each phase is independently shippable. W1 alone makes the app usable on
  desktop. W2 alone improves both platforms. Nothing breaks mid-migration
  because every phase is additive — the Android path continues to work
  unchanged throughout.

W1 — PLATFORM BRIDGE (decouple JS from AndroidBridge)
───────────────────────────────────────────────────────────────────────────────

Goal: every feature that currently requires AndroidBridge works on desktop
via browser-native APIs. The Android path is unchanged — it still uses
the native bridge. Desktop gets its own implementation.

Architecture: a single `platform-bridge.js` module (new file in src/utils/)
exports a PlatformBridge object. At boot, it detects the environment and
wires up the correct implementation. All existing call sites change from
`window.AndroidBridge.foo()` to `PlatformBridge.foo()`.

Detection: `!!(typeof window !== 'undefined' && window.AndroidBridge)` —
truthy check matching the existing pattern at JournalRecordingSheet:214
verbatim. Per [[platformbridge-single-source]], this expression appears
in EXACTLY ONE place: platform-bridge.js. Consumers never check
window.AndroidBridge directly after W1.2.

  ─── DISPATCHER CONTRACT (single-source Escape, no listener races) ───
  Refinement from review: modals do NOT add their own Escape keydown
  listeners. If two listeners on the same target both react to Escape,
  the modal's listener fires first → calls dismiss → unregisters → app-
  level handler then reads isAnyOpen() as false → fires handleAndroidBack.
  One Escape press = dismiss AND navigate. Wrong.

  Resolution: the app-level Escape handler in (c) is the ONLY dispatcher.
  Registry entries store `{ id: string, dismiss: () => void }` — not just
  the ID. When Escape fires:
    • document.fullscreenElement → skip (browser native exit, no
      preventDefault)
    • registry.isAnyOpen() → call peek().dismiss(). Do NOT also call
      handleAndroidBack. The dismiss callback handles its own state
      teardown; the registry unregister fires from its unmount effect.
    • else → handleAndroidBack()

  The registry uses a Set<string> for ID lookup + a parallel Map<string,
  () => void> for dismiss callbacks (or equivalently a single
  Map<string, { id, dismiss }> with the Set behavior emergent — Sets
  in JS preserve insertion order, so peek() = "last registered"
  works naturally on Maps too).

  ─── CONFIRMSTRIP UNIQUE-ID CONTRACT ───
  Refinement from review: the original "registering once inside
  ConfirmStrip's body covers all 12 usage sites" wording was WRONG
  for concurrent instances. If a future screen mounts two ConfirmStrip
  instances simultaneously (a list with multiple delete-confirmable
  rows, for example) and both register under the shared key
  "confirm-strip", then when instance A unmounts and unregisters
  "confirm-strip", the Set drops the key — even though instance B
  is still rendering. The next Escape fires back-nav while B is mid-
  confirm.

  Resolution: each ConfirmStrip instance generates a UNIQUE registry ID
  via React.useId() (React 18 native; available in this codebase).
  Registration key is `'confirm-strip:' + reactGeneratedId`. Each
  instance's mount-effect registers + cleanup unregisters its own
  unique ID. Set + Map handle the multiple-instance case naturally;
  the change is one-line at the registration site, no architectural
  cost.

  Same principle applies to any other modal where multiple
  simultaneous instances are plausible. Single-instance overlays
  (welcome-modal, garden-warning) can use a literal string ID since
  only one is mountable at a time by construction.

  Triggers detail:

    1. Keyboard: keydown 'Escape' → handleAndroidBack() — GATED.
       Per [[escape-key-gating]], the listener fires ONLY if:
         - document.fullscreenElement === null  (not in fullscreen)
         - No app-managed modal/sheet open
       If either condition false: do NOT preventDefault — let the
       browser exit fullscreen / close its own dialog. Swallowing
       Escape would break the Garden full-screen view + every
       open-sheet close-on-escape pattern.

       Per [[centralized-modal-signal]], the "no modal/sheet open"
       check MUST go through a SINGLE centralized signal — NOT a
       multi-source state-check on AppShellSheets + AppShellOverlays
       + screen-local sheet states. Implementation: useModalRegistry
       hook backed by a Set<string> of modal IDs.
         - register(id: string) — Set.add(id) on sheet mount
         - unregister(id: string) — Set.delete(id) on sheet unmount
         - isAnyOpen() — registry.size > 0  (O(1))
         - openIds() — Array.from(registry)  (for debug logs)

       Set<string> beats a counter:
         - Same O(1) check.
         - When debugging, openIds() shows WHICH modals are open
           ("note-sheet, bookmark-popover") — infinitely more useful
           than "modal count: 2."
         - Immune to double-register/unregister bugs (Set is idempotent;
           a counter goes negative under React's double-invoke in
           StrictMode, then isAnyOpen() lies).

       PREREQUISITE — exhaustive modal inventory (see CHECKLIST below).
       The hook is ~20 lines; the wiring audit across every sheet /
       overlay / popover / screen-local modal is the real work. If
       even ONE is missed, Escape fires when it shouldn't and the user
       navigates away mid-action. Land the inventory in this PLAN
       BEFORE writing the hook, and use it as the checklist to ensure
       every modal calls register/unregister.

    2. Browser: History API popstate → handleAndroidBack(). REQUIRES
       the history-sync layer from step (b) — today's React-state-only
       navigation pushes ZERO history entries, so popstate would never
       fire without (b). The sub-step (b) implementation:

         A useEffect (in useNav or a dedicated useHistorySync hook)
         watches the active-tab nav-key tuple — screen + bookId +
         chapterNum + letterId + studyId + studyChapterId + genreId
         + gardenPage. On any change, call history.pushState(state,
         '', ''). The active tab's identity is the source; multi-tab
         is one stream of history entries from the user's perspective.

       pushState state-object: OPTION B LOCKED per [[root-of-history-pwa]].
         The state object holds NOTHING meaningful. popstate handler
         doesn't try to restore from event.state; it just calls
         handleAndroidBack which pops the internal nav stack.
         Trade-off: browser forward-button is dead (no state to
         forward to). But this matches Android behavior (no forward
         button there either), reduces risk, and avoids a second
         source of truth that could disagree with the in-memory tab
         state. Option A (full nav state in pushState, enables
         forward + direct restoration) is deferred to a later polish
         phase if anyone misses forward-button behavior.

       Per [[root-of-history-pwa]], the root-of-stack design is a
       REQUIRED double-tap-to-exit pattern (NOT an optional toast).
       PWA has no "minimize" equivalent — without the guard the first
       back-press past the home screen EXITS the PWA. Without the
       toast the user is silently TRAPPED. Both pieces are mandatory:

         First back at root:
           1. history.pushState({}, '', '') — replacement entry, app stays
           2. Show toast "Press back again to exit"
           3. _rootExitArmed = true; setTimeout(2000) to disarm
         Second back within 2s (while armed):
           1. _rootExitArmed = false; clearTimeout
           2. Skip the replacement push — let browser exit normally
         After 2s without second back:
           _rootExitArmed flips false; next first-back gets a fresh toast.

       CRITICAL: the armed timer MUST CLEAR on any forward navigation.
       Otherwise: user presses back at root → toast + timer armed →
       taps a tile (forward nav) → navigates deep → presses back twice
       to return to root → first back at root SKIPS the toast because
       the timer is still armed from the earlier cycle → user
       surprise-exits. Tie timer-clear to the SAME nav-key effect that
       step (b) watches; any nav-key change while armed = clear timer.

       Tests:
         - popstate at root → toast appears, app stays
         - popstate at root × 2 within 2s → second one allows exit
         - popstate at root → wait 3s → popstate again → fresh toast (cycle resets)
         - popstate at root → forward-nav → popstate at root again →
           FRESH toast (timer was cleared by the forward-nav)
         - popstate mid-stack → no toast, handleAndroidBack runs normally
         - history.pushState fires on every nav-key change (verify
           the sync layer)

  state, returns "true"/"false"). The trigger source changes; the
  return-handling for "false" diverges by platform (Android: minimize;
  Web: arm-double-tap-then-exit). Register listener in useAndroidBack.js
  (already owns this domain).

FONT TOGGLE — Classic / Modern (LANDED post-W1, pre-W2)
───────────────────────────────────────────────────────────────────────────────

Context: W0.1 fixed a long-standing latent bug — @font-face declarations had
pointed to nonexistent files (Cinzel-Regular.otf, EBGaramond-VariableFont_
wght.ttf) since they were first authored, so for the entire life of the app
on every platform, Cinzel + EB Garamond silently fell back to the device's
system serif. The "fix" in W0.1 (real WOFF2 files committed, @font-face
filenames corrected) was technically correct but introduced a user-visible
regression: EB Garamond has a ~15-20% smaller x-height than typical system
serif, and Cinzel is a thin display font at small sizes. The CSS font sizes
(1.08rem body, 0.78rem date, etc.) had been effectively calibrated against
the system serif fallback users were ACTUALLY reading for years. Once the
real fonts loaded, body text felt smaller, headings felt spindlier, and
readability dropped — especially for older users.

Decision: keep both, default to Classic, ship a toggle. Plan for a fully
adjustable font/size system later (the design opens that door cleanly).

  Implementation (current — minimal, ship-ready):
  ────────────────────────────────────────────────
  Approach: instead of replacing 260 hardcoded font-family declarations in
  app.css with CSS variables (high churn, high risk), the @font-face block
  itself is toggled. When disabled, every `font-family: 'Cinzel', serif` in
  app.css automatically falls back to `serif` via the browser's built-in
  font fallback — zero CSS rule changes required.

    index.html line 28: <style id="custom-fonts"> wrapping the 6
                       @font-face declarations. EB Garamond weight range
                       bumped from `400 500` to `400 800` so when Modern
                       is active, real bold (600+) renders instead of
                       browser-synthesized bold.
    index.html line 66: boot script reads vot-state.settings.fontStyle.
                       If not "modern", sets customFontsEl.disabled=true
                       BEFORE React mounts. No FOUC.
    use-settings.js:    body-class effect mirrors fontStyle onto
                       customFontsEl.disabled at runtime. Live toggle
                       works without reload.
    SettingsScreen.jsx: "Modern Fonts" SettingsRow in the Text & Translation
                       section, just under Bible Translation.
    Default:            fontStyle = "classic" (system serif — what every
                       user has been seeing for the life of the app).

  Future — widely adjustable fonts (post-W2 polish):
  ───────────────────────────────────────────────────

    - Multiple font choices (system serif, EB Garamond, Cinzel, sans-serif,

  W2.2 — Write-through cache layer (modify cached-store.js)
  ──────────────────────────────────────────────────────────
  Extend CachedStore to support an async backing store:

    _load()    → read from in-memory cache (sync, fast)
                  if cache is cold, return default (empty)
    _save()    → write to in-memory cache (sync, immediate)
                  ALSO write to IDB (async, with error handling —
                  every write wraps in QuotaExceededError catch →
                  StorageHealth.onWriteFailure)
    _hydrate() → async one-time load from IDB into memory
                  called at boot, before React renders

  The hydration gate: App() renders a loading screen until all stores
  report hydrated. This is the ONLY async boundary — after hydration,
  every read is sync from memory. Zero render-path jank.

  Boot sequence:
    1. index.html inline script reads `vot-state` from localStorage
       (sync) for theme class — KEEP THIS as the fast-path shim. The
       localStorage copy is updated on every _save() as a write-behind
       mirror for the next cold boot's theme shim.
    2. React mounts, App() calls hydrateAllStores() (async).
    3. Each store opens IDB, reads data, populates in-memory cache.
    4. Promise.all resolves. App() sets hydrated=true, renders.
    5. All subsequent reads are sync from memory.

  Hydration timeout — TWO data-loss vectors (per [[w2-hydration-data-loss]]):

  Vector 1 (late stomp): if IDB open takes >3 seconds, naive impl
  renders with empty cache + background retry. User writes A. Retry
  completes, assigns cache = loaded IDB → A is lost.

  Vector 2 ("cache wins" is the WRONG fix): if degraded-mode treats
  the cache as authoritative and an IDB retry's "cache wins on
  conflict" is interpreted as "discard IDB, keep cache" — that
  destroys ALL previous-session data. Cache during degraded is {A};
  IDB has {B, C, D}; "cache wins" → keep {A}, lose {B, C, D}.
  Catastrophic.

  Required design — write-queue REBASE (NOT a store-level cache-wins merge):

    State machine per store: 'pending' | 'loaded' | 'degraded'.

    'pending' (boot → hydration in flight, ≤3s):
       Reads return defaults (cache is empty).
       Writes go to a per-store WRITE QUEUE — separate from cache; the
       cache stays at defaults.
       UI shows a brief "Loading..." indicator on the affected surfaces.

    'loaded' (hydration succeeded within timeout):
       1. Cache = loaded IDB data (the base — previous-session truth).
       2. REPLAY the write queue on top — each queued write applied to
          cache; queue wins on per-key conflicts since current-session
          writes are newer.
       3. Flush rebased cache → IDB so post-replay state is durable.
       4. Drain queue. Transition to 'loaded'.
       5. Subsequent writes are normal write-through.

    'degraded' (hydration timeout, e.g. 3s elapsed):
       Mark store degraded.
       KEEP queue separate from cache — DO NOT drain queue into cache
       as if it were authoritative. (That's the trap that leads to
       "cache wins discards IDB.")
       Reads serve `defaults + appliedQueue(key)` — i.e., the queue is
       the source of truth for keys it touched; defaults for everything
       else. (Implementation: virtual overlay of queue on defaults, or
       a read helper that consults queue first.)
       Writes: still go to queue.
       UI warning: "Storage temporarily unavailable; your latest
                    changes are kept in memory and will save when
                    storage recovers."
       Background IDB retries continue.

    'degraded' → IDB eventual recovery:
       1. Cache = loaded IDB data (the previous-session base).
       2. REPLAY entire write queue on top of cache.
       3. Flush rebased cache to IDB.
       4. Transition to 'loaded'.
       5. NO DATA LOSS: IDB provides previous-session data; queue
          provides current-session writes. Both survive.

    Persistent failure (IDB never recovers — corrupted DB, permanent
    permission denial): store stays 'degraded' forever (until app
    reload). Queue is the ONLY truth for the session. App remains
    usable; warning persists.

  CRITICAL wording for code/tests/docs: "REBASE the queue ON TOP OF
  IDB" — NOT "merge with cache wins on conflict." The latter is
  ambiguous and can be implemented destructively. The former is
  unambiguous. The W2.2 agent MUST use this terminology and MUST NOT
  implement a store-level "keep cache, discard IDB" merge.

  Tests (must exist in W2.2 vitest):
    1. Hydration <100ms → 'loaded' → write-through works normally.
    2. Hydration 100ms-3s + writes during 'pending' → both previous-
       session IDB data AND current-session queue writes survive.
    3. Hydration >3s + writes during 'pending' + IDB eventually
       resolves: BOTH previous-session IDB data {B,C,D} AND
       current-session writes {A} present in final cache. Test
       assertion: `cache.keys() ⊇ {A,B,C,D}` and `cache.A === A`.
    4. Hydration NEVER completes (corrupted DB / permission denied) →
       store stays 'degraded'; app remains usable; queue is source of
       truth; warning surfaced.
    5. Multiple stores hydrate concurrently — some succeed, some time
       out. Each store's state tracked independently.

  Per [[user-data-paramount]], silent stomping is unacceptable; the
  W2 agent MUST implement the rebase semantics correctly before any
  consumer migrates to the async-hydrated path.

  W2.7 — Storage Health system (dynamic, platform-aware user warnings)
  ────────────────────────────────────────────────────────────────────
  The app reads the storage situation continuously and communicates
  risks to the user in plain language — before they lose data, not
  after. No jargon. Every warning tells the user what to DO. Warnings
  are invisible when everything is fine.

  Architecture: StorageHealth module (src/utils/storage-health.js) owns
  detection, assessment, and state. UI components subscribe to it.

  Detection engine — StorageHealth.assess() runs:
    1. Platform detection (one-time, cached):
       Android WebView / Safari tab / Safari PWA / Firefox / Chrome /
       private browsing (probe via IDB open + quota check)
    2. Quota snapshot (async): navigator.storage.estimate()
    3. Persistence check: navigator.storage.persisted()
    4. Derived health tier:
       HEALTHY  — <50% used, persisted, no platform risks
       CAUTION  — 50-80% used, OR not persisted, OR Safari tab
       WARNING  — 80-95% used, OR quota < 100 MB
       CRITICAL — >95% used, OR quota < 50 MB, OR private browsing
       READONLY — active write failure this session (QuotaExceededError)
    5. Risk flags (zero or more): safari-7day, ios-pwa-isolate,
       low-quota, critical-quota, not-persisted, private-mode,
       write-failed, quota-declining

  StorageHealth API:
    assess()                → Promise<StorageHealthReport>
    checkBeforeWrite(bytes) → { ok, warning? }
    onWriteFailure(error)   → updates tier to READONLY
    onWriteSuccess()        → clears READONLY if quota recovered
    subscribe(callback)     → unsubscribe function
    getReport()             → last cached report (sync, for render)
    requestPersistence()    → Promise<boolean>

  When to check:
    1. Boot (after hydration)
    2. After every meaningful data creation (journal, annotation, etc.)
       — only re-assess if tier >= CAUTION
    3. Periodic: every 5 min while app is active (disk space changes)
    4. After any write failure: immediate re-assess
    5. On resume from background (visibilitychange): if >60s stale

  W7.2 — raw() immutability (store mutation safety)
  ─────────────────────────────────────────────────
  Problem: `CachedStore.raw()` returns the live `_cache` object. A
  consumer can mutate the returned object without calling `_save()`,
  and the change persists in memory but never reaches IndexedDB. This
  is a silent-corruption vector.

  Current state: research found only 1 call site (a test), and zero
  production mutations via raw(). But the API PERMITS it, and new code
  could introduce the bug unknowingly.

  Fix: `raw()` returns `Object.freeze()` of the cache (shallow freeze).
  Consumers that need to mutate must go through the store's named methods
  (add/remove/update/set) which call `_save()` + `_bump()` internally.

  Why Object.freeze, not structuredClone:
    - Freeze is O(1) (just sets a flag on the object), clone is O(n).
    - Annotations store can be 500 KB — cloning on every read is too
      expensive for the render-path hot stores.
    - Freeze catches accidental mutation at dev time (throws in strict
      mode, silent no-op in sloppy mode). Good enough — strict mode is
      default in ES modules.

  Implementation:
    - cached-store.js: change `raw() { return this._load(); }` to
      `raw() { const d = this._load(); Object.freeze(d); return d; }`
    - Stores that return nested objects (annotations has hlKey → Array):
      freeze the top level only. Deep-freeze is too expensive and
      unnecessary — named methods are the write path.
    - Run all 476 vitest tests — any test that mutates via raw() will
      throw, exposing the violation.

  Tests:
    - New vitest case: `raw()` returns frozen object, mutation throws
    - Existing tests pass (proves no production code mutates via raw())

  W7.6 — OPFS upgrade for JournalMediaStore (perf optimization)
  ─────────────────────────────────────────────────────────────
  Problem: JournalMediaStore uses IndexedDB Blob storage (correct,
  GB-scale, works on all platforms). But IDB Blob writes for 100 MB
  payloads take ~850 ms vs ~90 ms with OPFS. For power users with
  hundreds of recordings, the cumulative I/O overhead matters.

  OPFS advantages: 3-4x faster writes, partial range reads via
  `read(buffer, { at: offset })` without loading whole file, no
  structured clone overhead.

  OPFS constraints:
    - Fast sync API (createSyncAccessHandle) ONLY works in a Web Worker
    - Safari has known bugs (WebKit #250495 — data loss on handle create)
    - Firefox: OPFS-only, no full File System Access API
    - Async-only API on main thread (no sync reads)

  Architecture: dedicated BlobWorker (src/utils/blob-worker.js).
  Main thread communicates via postMessage. JournalMediaStore API
  stays identical — the change is the backing store, not the contract.
  Fallback: if OPFS unavailable or fails write-read-verify test on
  first launch, stay on IDB Blobs (current behavior).

  Gate: profile JournalMediaStore I/O under real usage BEFORE building
  this. If 90th percentile write latency is <200ms (likely for typical
  memo sizes of 50-500 KB), the optimization isn't worth the Worker
  complexity + Safari risk. Build only if profiling justifies it.

  W8.2 — CSS @layer architecture (eliminate all !important)
  ─────────────────────────────────────────────────────────
  Problem: 24 !important declarations remain in app.css. Research
  classified them:
    - 11 annotation/highlight display (prevent palette bleed)
    - 6 light-mode palette guards (state-specific overrides)
    - 2 visibility/drag states
    - 5 structural/spacing overrides

  All 24 are load-bearing — they exist because the cascade order of
  existing rules doesn't match the intended priority. Removing them
  without restructuring the cascade would break visual output.

  Fix: CSS @layer (supported in all browsers since 2022, Safari 15.4+,
  Chrome 99+, Firefox 97+). Define explicit cascade layers:

    @layer reset, base, layout, components, annotations, states, overrides;

  Rules in later layers win over earlier layers regardless of
  specificity. This means:

    - `@layer annotations` contains .hl-note, .hl-underline, per-color

W9 — DATA INTEGRITY (schema validation + import hardening)
───────────────────────────────────────────────────────────────────────────────

Goal: eliminate the "data fragility" weakness entirely. Today's gate
(check_balance.py + esprima) proves "valid JavaScript" but says nothing
about "valid letter." A file with a missing `id`, an unknown block type,
an orphaned footnote reference, or a broken prev/nextLetter chain passes
every current check. Schema validation closes that gap at build time AND
at import time.

Current validation gap:
  check_balance.py catches:
    - Brace/bracket/paren imbalance
    - Non-ASCII dashes in verse ranges (en/em dash between digits)
    - Smart quotes used as JSON delimiters
    - esprima JS parse errors (real syntax check)
  check_balance.py does NOT catch:
    - Missing required fields (id, title, blocks, paragraphs)
    - Invalid block types (typo "parra" instead of "para")
    - Invalid segment types (typo "itaric" instead of "italic")
    - Orphaned footnote refs (fn "3" in blocks but no "3" in footnotes)
    - Orphaned nkjv keys (scripture ref in nkjv but no footnote cites it)
    - Broken prev/nextLetter chains (letter A says next is B, but B says
      prev is C)
    - Invalid paragraph align values (not "center"|"justify"|"left")
    - Missing scriptures for {{ref:...}} inline refs in Format B text
    - Wrong types (num as string instead of number, date as number)

  W9.1 — Schema definitions (4 corpus format validators)
  ──────────────────────────────────────────────────────
  New file: tools/validate-schemas.js (Node.js script).

  Each schema is a validator FUNCTION, not a JSON Schema document. JSON
  Schema can't express cross-field constraints (footnote refs must exist
  in the footnotes object; nkjv keys must be cited by at least one
  footnote). Functions can. The validators return an array of violation
  strings — empty array = valid.

  Format A validator (Volumes, Lord's Rebuke, Letters families, Hidden Manna):
    Required top-level: id (string), title (string), blocks (array)
    Required per-letter: footnotes (object), nkjv (object)
    Optional: num (number), date (string), from (string), spoken (string),
              forLine (string), audioUrl (string), soundcloudUrl (string),
              videoVoiceUrl (string), videoMusicUrl (string),
              relatedTopics (array of {label, url}), metaAddendum (string),
              metaAddendumUrl (string), metaAddendumLink (object),
              metaAddendumInternal (string), prevLetter (object|null),
              nextLetter (object|null)
    Block validation:
      - type must be one of: para, poetry, closing, closing-fn, note, scripture
      - para/closing-fn: must have segments (array of {t, v})
      - poetry: must have lines (array of arrays of {t, v})
      - closing/note: must have text (string)
      - scripture: validated but structure varies
    Segment validation:
      - t must be one of: text, italic, bold-italic, caps, fn, stanza-break,
        letter-link
      - fn segments: v must exist as a key in the letter's footnotes object
      - stanza-break: v is optional (usually absent)
    Footnote validation:
      - Every footnote key must be referenced by at least one fn segment
        (orphan detection — warns, doesn't fail)
      - type must be "scripture" or "note"
      - scripture type: ref (string) must exist as a key in nkjv
      - note type: text (string) required
    nkjv validation:
      - Every nkjv key should be referenced by a footnote with type
        "scripture" (orphan detection — warns, doesn't fail)
    Chain validation (prev/nextLetter):
      - If prevLetter is not null, it must have id (string) and title (string)
      - If nextLetter is not null, same
      - Cross-letter: letter[i].nextLetter.id === letter[i+1].id AND
        letter[i+1].prevLetter.id === letter[i].id (chain consistency)

  Format B validator (WTLB One, WTLB Two, The Blessed, Holy Days):
    Required: id (string), title (string), paragraphs (array)
    Required per-paragraph: align (one of "center"|"justify"|"left"),
                            text (string)
    Optional: num (number), scriptures (object), sourceLabel (string),
              type (string), prevEntry/nextEntry (object|null)
    Inline ref validation:
      - Every {{ref:Book Ch:V}} in text must have a matching key in the
        entry's scriptures object (if scriptures exists)
      - Every {{nav:bookId:chapter}} is syntactically valid (2 colon-
        separated parts)
    Chain validation (prev/nextEntry):
      - Same consistency check as Format A chains

  Format C validator (Bible books — books.js, books-restored.js):
    Required: id (string), title (string), chapters (array)
    Per-chapter: num (number), sections (array)
    Per-section: heading (string, can be empty), verses (array)
    Per-verse: n (number), text (string)
    Verse numbering: n values within a section must be ascending and
    contiguous (gaps indicate data corruption)

  Format D validator (Bible Studies — bible-studies.js):
    Required per study: id (string), title (string), parts (array)
    Per-part: title (string), chapterIds (array of strings)
    ChapterIds: each must reference a valid chapter entry in the study's
    chapters array

  Implementation:
    - tools/validate-schemas.js exports: validateFormatA(letters, opts),
      validateFormatB(entries, opts), validateFormatC(books, opts),
      validateFormatD(studies, opts)
    - Each returns { errors: string[], warnings: string[] }
    - Errors = hard failures (build should fail)
    - Warnings = suspicious but possibly intentional (orphan nkjv keys,
      unusual structures)
    - opts.strict = true for CI/build; false for exploratory checks

  Tests:
    - vitest: valid letter passes, each violation type detected
    - vitest: chain consistency across a 3-letter array
    - vitest: orphan detection for footnotes + nkjv

  W9.2 — Build-time validation (integrate into build pipeline)
  ────────────────────────────────────────────────────────────
  Extend tools/build.py (or add a parallel Node.js build step) to run
  schema validation on every data file BEFORE concatenation.

  Two approaches (choose one):

  Option A — Node.js pre-step:
    New npm script: "validate:data" → node tools/validate-schemas.js
    The script loads each data file via vm.runInNewContext(), captures the
    global variable, runs the appropriate format validator.

    Con: maintaining parallel JS/Python validators; Python can't import
    the data files natively.

  Recommendation: Option A. The data files are JavaScript; the validators

  W9.3 — Import payload validation (V2 schema + per-store validation)
  ──────────────────────────────────────────────────────────────────
  Problem: _doImport in SettingsScreen.jsx validates only `app ===
  'VOTReader'` and checks exportVersion. The store data payloads are
  written to IDB without any structural validation. A corrupted or
  maliciously crafted export file can write arbitrary shapes into stores,
  causing runtime crashes on next read.

  Fix: validate each store's payload before writing. Three layers:

  Layer 1 — Export envelope validation:
    - app === 'VOTReader' (already exists)
    - exportVersion is 1 or 2 (already exists)
    - data is object (LS keys)
    - stores is object (IDB store payloads)
    - media is object or undefined (journal blobs)
    - No unexpected top-level keys (forward-compat: warn, don't fail)

  Layer 2 — Per-store shape validation:
    New utility: src/utils/import-validators.js
    Each store gets a lightweight validator that checks the shape of its
    data BEFORE replaceAll/setAll/set is called:

    AnnotationStore: value is object, each key maps to array of
      { hlKey, startOffset, endOffset, color, style, ... }
    NoteStore: value is object, each key maps to { text, color, ... }
    BookmarkStore: value is object, each key maps to
      { bookId, chapterNum, label, ... }
    LinkStore: value is object, each key maps to array of
      { source, target, ... }
    JournalStore: value is object, entries have { id, blocks, ... }
    JournalIndexStore: value is object with known fields
    NotebookStore: value is object, each notebook has { id, name, ... }
    StateStore: value is object with settings sub-object
    RecentNavStore: value is array of nav-history records
    HomeOrderStore: value is array of strings
    ProphecyCardsStore: value is object of boolean states
    Flag stores (3): value is truthy/falsy

    Validation depth: check top-level shape and required field presence.
    Do NOT deep-validate every annotation — that's too expensive for
    import and the data was valid when exported. Check that the structure
    is plausible, not that every field is semantically correct.

  Layer 3 — Per-store error isolation:
    Change _doImport to wrap each store's write in try/catch:
      for (const [storeName, payload] of Object.entries(stores)) {
        try {
          const violations = validateStorePayload(storeName, payload);
          if (violations.length) {
            importWarnings.push(`${storeName}: ${violations.join(', ')}`);
            continue;  // skip this store, don't write corrupted data
          }
          await targetStore.replaceAll(payload);
          importedCount++;
        } catch (err) {
          importFailures++;
          failedStores.push(storeName);
        }
      }

    After all stores processed, show a detailed toast:
      - "Imported 12/14 stores. 2 skipped (validation failed: annotations,
        links). See export for backup."
      - If ALL stores fail: "Import failed — no data was changed. The
        file may be corrupted."

  Layer 4 — Media blob validation:
    For each media entry in the import payload:
      - key must be a string (blob ID)
      - value must be a valid base64 string (regex check on first 100
        chars — full decode happens during write)
      - Decoded size must be < 100 MB per blob (matches existing guard)

  Implementation:
    - New file: src/utils/import-validators.js (~200 lines)
    - Modify: SettingsScreen.jsx _doImport (~50 lines changed)
    - Tests: vitest for each store validator (valid passes, each
      violation type detected, malformed payloads rejected)

  W10.5 — Motion and visual accessibility
  ───────────────────────────────────────
  Respect user preferences for reduced motion, high contrast, and
  focus visibility.

  W10.5a — prefers-reduced-motion:
    Current state: ZERO references in the codebase. All CSS animations
    (sheet slide-up, overlay fade-in, tab transitions, toast appear)
    run unconditionally.

    Add to app.css:
      @media (prefers-reduced-motion: reduce) {
        *, *::before, *::after {
          animation-duration: 0.01ms !important;
          animation-iteration-count: 1 !important;
          transition-duration: 0.01ms !important;
          scroll-behavior: auto !important;
        }
      }

    This is the nuclear option — kills ALL animations for users who
    prefer reduced motion. It's the safest starting point because:
      1. It's one rule, not 30 per-animation overrides
      2. It respects the user's system setting unconditionally
      3. It can't miss a new animation added later
      4. The 0.01ms (not 0ms) preserves animation-end events that
         JS code may depend on (prevents logic breakage)

    If specific animations should survive for reduced-motion users
    (e.g., a subtle opacity fade is acceptable), carve out exceptions
    AFTER the nuclear rule lands and is verified.

  W10.5b — Focus-visible indicators:
    Current state: browser defaults. Some browsers show outlines on
    click (annoying for mouse users) and others suppress them entirely
    (invisible for keyboard users).

    Add to app.css:
      :focus-visible {
        outline: 2px solid var(--hl-yellow);
        outline-offset: 2px;
      }
      :focus:not(:focus-visible) {
        outline: none;
      }

    This shows outlines ONLY for keyboard navigation (not mouse clicks).
    The outline color uses the existing --hl-yellow var for visual
    consistency with the annotation system. Dark mode: --hl-yellow is
    already visible on dark backgrounds. Light mode: same.

  W10.5c — Forced colors / high contrast:
    Add to app.css:
      @media (forced-colors: active) {
        .hl-note, .hl-underline { forced-color-adjust: none; }
      }

    This preserves annotation colors in Windows High Contrast mode.
    Without it, all highlight colors collapse to the system highlight
    color (usually blue), making the 10-color palette meaningless.

    Other elements (buttons, nav, text) should FOLLOW forced-colors
    to respect the user's contrast needs. Only annotation highlights
    opt out because color IS the information.

RISK REGISTER
───────────────────────────────────────────────────────────────────────────────

  HIGH — IndexedDB sync render-path (W2 Tier 3)
    The 5 stores read synchronously in render paths (annotations, notes,
    links, state, prophecy-cards) must be in-memory before React renders.
    The hydration gate is the mitigation, but boot-time latency may
    increase. Measure: time from page load to first render, compare
    pre/post migration. Target: < 200ms overhead.

  HIGH — Safari 7-day data eviction (W2.7 Scenario 3)
    Safari evicts all script-writable storage after 7 days without user
    interaction. Home-screen PWAs are exempt. Users who create data in
    a Safari tab without installing will lose everything. Mitigation:
    intercept first data-creating gesture with a modal explaining the
    risk + install instructions. Can't fix the platform constraint —
    can only warn clearly enough that the user acts before data loss.

  MEDIUM — Export/import media round-trip (W2.6)
    exportPersonalData currently drops all journal media blobs (audio,
    images). Fix adds async base64 encoding of all blobs into the JSON
    payload. For users with >100 MB of media, this may OOM or produce
    a multi-hundred-MB JSON file. Mitigation: 100 MB guard warns user;
    ZIP-based streaming export deferred to W7.

  MEDIUM — iOS PWA storage isolation (W2.4 + W2.7 Scenario 4)
    Installing a PWA on iOS creates a NEW storage container. Data from
    the Safari tab is invisible. Users who build up data in Safari then
    install lose access to it. Mitigation: detect empty-storage-on-
    first-PWA-boot and show an export/import bridge card. Can't fix
    the platform constraint — Apple isolates by design.

  MEDIUM — Budget Android near-zero quota (W2.7 Scenario 7)
    Chrome/WebView quota = 60% of FREE disk. Devices with <500 MB free
    get quotas as low as a few MB. App must detect and warn, not crash.
    Mitigation: startup health check + QuotaExceededError handling on
    every write path → graceful degradation to read-only mode.

  MEDIUM — Audio recording cross-browser (W1.4)
    MediaRecorder API: Chrome supports audio/webm, Firefox supports
    audio/ogg. Safari doesn't support MediaRecorder reliably. Need
    to test mime negotiation and base64 encoding on all three browsers.
    Mitigation: mime detection at start time, fallback to audio/wav
    if needed.

  MEDIUM — Service worker cache size (~12.5 MB total)
    Full corpus cache is large. Some browsers may throttle or reject.
    Mitigation: cache critical path on install (~2.3 MB), lazy bundles
    on first fetch. Monitor navigator.storage.estimate() for pressure.

  MEDIUM — hlTick removal blast radius (W7.3)
    39 files touched. The 8 truly-live uses must be migrated to
    useSyncExternalStore before the plumbing can be removed. If any
    live use is missed, annotation re-renders will silently stop
    working. Mitigation: smoke walk every annotation operation (add,
    remove, color change, note create, bookmark toggle, link add)
    after W7.3a before starting W7.3b.

  MEDIUM — CSS @layer cascade rewrite (W8.2)
    Moving 4,100 lines of CSS into @layer blocks changes the cascade
    globally. Unexpected interactions could break any visual state.
    Mitigation: scaffold step (W8.2a) wraps without removing any
    !important; breakage from layer order itself is caught before
    any rule changes. Screenshot comparison at each step.

  LOW — CSP compatibility on GitHub Pages
    'self' should resolve correctly. If not, fall back to explicit
    origin in CSP. Easy to diagnose via console errors.

  LOW — Font licensing
    Cinzel + EB Garamond are OFL (SIL Open Font License). Bundling
    in an APK and serving via GitHub Pages are both permitted uses.
    No license risk.

  LOW — UI typing flood (W8.1)
    82 files × ~5 type errors each = ~400 initial errors. But Q4
    proved the pattern: file-by-file, 4-7 files/day, zero-tolerance
    after each. Risk is tedium, not breakage.

  MEDIUM — Schema validation false positives (W9.1-W9.2)
    The 21 data files evolved organically over months. Some structural
    patterns may be intentional but look like violations to a strict
    schema (e.g., Hidden Manna has only 1 letter, Holy Days entries
    borrow structure from Format B but have extra fields like
    sourceLabel). Mitigation: run validators against ALL current data
    files BEFORE declaring schemas final. Any false positive = schema
    bug, not data bug. warnings vs errors distinction ensures
    intentional deviations don't block the build.

  LOW — Import validation rejects valid V1 exports (W9.3)
    Users with pre-W2 V1 export files import through a backward-compat
    path that parses LS JSON strings. The shape validators must handle
    V1's stringified format, not just V2's native objects. Mitigation:
    validators run AFTER the V1→V2 parsing step, not before. Test with
    a real V1 export payload.

  MEDIUM — Focus trap interactions with existing keyboard handling (W10.2)
    The app already has: useModalRegistry (Escape dispatch), history-sync
    (popstate), Escape key listener (6 gates), ConfirmStrip registration.
    Adding focus traps creates a new keyboard interceptor that must
    coexist. Specifically: Tab/Shift-Tab trapping in a sheet must NOT
    interfere with Escape dismissal (different keys, should be fine) but
    MUST handle the case where ConfirmStrip appears inside a trapped
    sheet (the trap's focusable-element query must re-run when the
    strip mounts/unmounts). Mitigation: pilot on NoteSheet (has
    ConfirmStrip + textarea + mode transitions) before rolling out.

  LOW — Heading hierarchy retrofit (W10.4b)
    Changing screen titles from <div>/<span> to <h1> may affect CSS
    rules that target element type (e.g., `div.screen-title`). Audit
    finds: screen titles are styled via class selectors, not element
    selectors. Low risk. Mitigation: visual smoke walk after each
    screen's heading change.
