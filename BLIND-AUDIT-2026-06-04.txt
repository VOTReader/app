--------------------------------------------------------------------------------

[ ] BLD-1  [M](V)  Web deploy publishes to production with ZERO build gates  (S)
    Where: .github/workflows/deploy-web.yml:34-48,67-118
    Problem: deploy triggers on push:main with no workflow_run/needs/if gating on
      ci.yml and invokes 0 gate commands. A green `npm run build` that violates a
      DIFFERENT gate ships to the live PWA. Sharpest case: a corpus edit without a
      CORPUS_VERSION bump fails CI but NOT deploy → every installed PWA stuck on
      stale scripture forever.
    Fix: gate the deploy job — `on: workflow_run: {workflows:[CI], types:[completed],
      branches:[main]}` + `if: github.event.workflow_run.conclusion == 'success'`;
      OR inline the cheap checks as deploy steps:
      `npm run check:csp && node tools/check-corpus-version.js --check && npm run validate:data`;
      OR require the `build` check in branch protection.

[ ] BLD-2  [M](V)  APK packages committed dist/ with no build-time freshness check (M)
    Where: app/build.gradle.kts (no asset-regen task)
    Problem: AGP packages app/src/main/assets/ (incl. committed bundles) as-is;
      nothing runs tools/build.py / npm run build before assembling. A local edit +
      `gradle assembleDebug` without a clean commit ships an APK whose bundles don't
      reflect the edit; CI catches staleness only later, after the wrong APK may be
      sideloaded.
    Fix: add a gradle `preBuild.dependsOn` task that shells
      `python tools/build.py && npm run build:b/c/d/e/css`; OR at minimum a verify
      task that fails assembly when `git diff --quiet app/src/main/assets/dist` is dirty.

[ ] BLD-3  [L](V)  minify-bundle.mjs emits a 0-byte file on empty input, exits 0  (1line)
    Where: tools/minify-bundle.mjs:27-34
    Problem: empty input → exit 0, 0-byte output, no guard. build.py only checks
      returncode. A truncated/emptied source would silently ship an empty corpus
      bundle (only backstop is the corpus-version CHANGE detector, not emptiness).
    Fix: after buildSync, assert statSync(file).size > 0 (ideally > 50% of input for
      corpus bundles) and process.exit(1) otherwise. (Broken SYNTAX is already
      handled — esbuild exits 1 and leaves the input uncorrupted.)

[ ] BLD-4  [L](V)  Stale "WV69 floor" comments post-date the chrome69→108 lift  (S)
    Where: backup-container.js:74, platform-bridge.js:141-143, tools/build.py:9

    Problem: comments mislead a future reader about the real floor; behavior is correct.
    Fix: sweep chrome69 / WV69 / WebView-69 / WV-69 comment references → chrome108.

[ ] BLD-5  [L](V)  Raw-shipped assets ship un-minified (size win)  (S)

[ ] PERF-1  [H](V)  html2canvas(document.body) rasterizes the entire scrolled chapter  (M)
    Where: platform-bridge.js:550-562 (webTakeScreenshot) + use-thumbnails.js:176,211
    Problem: passes document.body at scale:min(devicePixelRatio,2) → a canvas of
      bodyWidth x fullScrollHeight x scale^2 x 4 bytes + a cloned DOM. Long chapter (Ps 119
      ~950 elements, body 5000-8000px) at 2x → ~10000px-tall offscreen canvas, tens-to->100
      MB transient. The maxDim:1440 downscale happens AFTER the full-res canvas exists.
      Fires on scroll-stop (300ms) AND 350ms after every nav → recurs constantly while
      reading. (Android unaffected — native PixelCopy captures only the viewport.)
    Fix: cap html2canvas working size up front (windowWidth/windowHeight = viewport, or
      render at scale:1 then downscale); OR skip the after-nav capture and only capture on
      an explicit tab-overview open.

[ ] PERF-2  [H](V)  Garden decoded-bitmap residency huge at Native/Ultra tiers  (S)
    Where: garden.js:52 (GARDEN_CACHE_MAX=12), :130-134 (new Image() per page) +
      GardenView.jsx:96-102 (preload current+5) + :108-144 (background crawler preloads
      every page of the tier)
    Problem: the cap bounds COUNT at 12, but a decoded bitmap is full JPEG resolution in
      RGBA regardless of display size: Standard 2880px ≈ 44 MB each → 12 ≈ 525 MB; Ultra
      4600px ≈ 112 MB each → 12 ≈ 1.3 GB. The crawler keeps warming pages; current+5
      forces 6 simultaneous decodes. On a 2-3 GB device Native/Ultra OOM/thrash.
    Fix: scale GARDEN_CACHE_MAX down by tier (12 mobile, 4 native, 2 ultra); gate Ultra/
      Native behind navigator.deviceMemory; reduce the +5-ahead preload to +1/+2 on high tiers.
