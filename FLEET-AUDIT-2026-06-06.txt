WHAT THIS IS. On 2026-06-06 a fleet of 16 independent Opus-4.8 agents
adversarially audited VOTReader, CODE ONLY (no UI/preview/device walks, no
accessibility, no splash/visual). Each agent owned one subsystem, verified every
finding against live source (several reproduced bugs through production code or
ran the suite), and returned a 1-10 rating + ranked findings. This file is the
canonical tracker for closing them. The full per-agent reports (more LOW/INFO
detail than is transcribed here) are in the session transcript.

OVERALL: 8.2/10. An exceptionally well-engineered personal app — visibly the
product of prior audit passes. The data-lifeline subsystems (journal durability,
security, backup, storage) are excellent. The gap to 9+ is a small, specific,
mostly-mapped set of fixes, several found by TWO agents from different angles.

LOAD-BEARING GOTCHAS
  - Preview clean-slate to load fresh bundles past the SW cache (unregister SWs +
    delete caches + reload). To verify SEARCH you must ALSO
    indexedDB.deleteDatabase('vot-search-cache').
  - jsdom has no navigator.locks -> vitest.setup.js ships a mutex shim. It is
    faithful; it does NOT mask STOR1 (that's a merge-ALGORITHM flaw).
  - A new window-globalized export needs `npm run lint:globals` + re-stage
    (pre-commit auto-regenerates now, but confirm).
  - CORPUS_VERSION manual bump on any books/matthew/VOT *and* (newly) on
    bible-studies.js if it enters the SW corpus cache (SW1).
  - app.jsx is under an ENFORCED <=800-line canary. Prefer pushing new state/
    effects into hooks over bloating App().
  - esbuild --target=chrome108 on build:b/c/d/e/css + corpus minify (Permanent
    Rule 6). Floor is a RUNTIME-API contract, not just syntax.
