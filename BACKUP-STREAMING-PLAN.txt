 BACKUP-STREAMING-PLAN.txt — GB-scale, enterprise-grade Export / Import
═══════════════════════════════════════════════════════════════════════════
 Created 2026-06-02. Strategic working memory for the export/import
 re-architecture. Owner directive (2026-06-02):

   "User should be able to accumulate, export and import GBs of data — as much
    as their device can store — safely, efficiently and quickly. Someone could
    feasibly use the Journal for years and accumulate tons of text, inserted
    images and audio recordings. That MUST be accounted for in a professional
    and extremely robust manner ... keep exporting and importing impeccable and
    easy. Enterprise level type behavior."

 This SUPERSEDES AUDIT-PLAN.txt [S4] ("lower the export ceiling") — lowering is
 a regression. It also REVISITS [S5] (the 100 MB aggregate import cap added
 2026-06-02): once import streams, that cap is removed/raised (it exists only
 because today's decode holds everything in heap).

───────────────────────────────────────────────────────────────────────────
 WHY THE CURRENT DESIGN CANNOT DO GBs (verified against source, 2026-06-02)
───────────────────────────────────────────────────────────────────────────
 The backup is ONE big JSON object with every media blob base64-encoded inline,
 produced + consumed whole. Every OOM point (file:line from the architecture map):

 EXPORT
  • buildExportPayload assembles ALL stores + ALL media base64 into one payload
    object (backup.js:230-241); each media blob → base64 string accumulated in
    `media{}` (backup.js:204).
  • JSON.stringify(payload) → ONE giant string (SettingsScreen.jsx:441).
  • Web: new Blob([json]) copies it again (platform-bridge.js:338).
  • Android: the giant string crosses the JS→native bridge as a single argument
    (AppInterface.saveToFile, :249) — a multi-GB string marshalled across the
    WebView bridge is the hard OOM wall — then content.toByteArray copies it
    again (StorageManager.kt:66).
 IMPORT
  • Android: readUriAsBase64 reads the WHOLE file into a ByteArray
    (StorageManager.kt:43-45) then Base64.encodeToString (+33%, :46), crosses the
    bridge as one string, JS atob's it (SettingsScreen.jsx:590), JSON.parse's it
    (:476). 4-5 full copies of a GB file.
  • Web: FileReader.readAsDataURL(whole file) (platform-bridge.js:315) → base64 →
    atob → JSON.parse. Same multiplicity.
  • base64ToBlob per record decodes inline (backup.js:69-82).

 Net: base64 inflates the file 33% AND the whole thing lives in heap several
 times over, on both platforms. Caps (50 MB import file / 100 MB export media)
 exist precisely to dodge this OOM — they are the symptom, not the cure.

───────────────────────────────────────────────────────────────────────────
 TARGET ARCHITECTURE — stream everything, never hold the whole payload
───────────────────────────────────────────────────────────────────────────
 Two independent problems, both must be fixed:
   (1) FORMAT: stop base64-inlining media into one JSON. Separate the small
       structured data (stores = JSON, bounded to MBs even after years of text)
       from the large binary media (images + audio = the GB-scale part), and
       store media as RAW bytes (no +33% base64).
   (2) I/O: read/write the file as a STREAM on both platforms so peak memory is
       ONE media blob (a few MB), never the whole backup.

 CHOSEN CONTAINER FORMAT (revised 2026-06-02 — see "format pivot" below):
   a bespoke LENGTH-PREFIXED binary container (exportVersion 3). On-disk
   (all integers big-endian):
     [8]            magic = "VOTBACK1" (ASCII)
     [8]            manifest length, bytes (uint64)
     [manifestLen]  manifest, UTF-8 JSON = { app:"VOTReader", exportVersion:3,
                      exportDate, diagnosticLog, storageQuota/Used, counts,
                      data:{vot-state:...}, stores:{...all IDB stores...},
                      media:[ {id, type, mime, size, width, height, duration,
                               created} ... ]  ← METADATA ONLY, NO bytes }
     for each media[i] in manifest.media, IN ORDER:
       [8]          blob length, bytes (uint64)
       [blobLen]    blob RAW bytes (no base64)

   FORMAT PIVOT — why NOT a standard zip (the plan's first instinct):
    A standard zip caps the TOTAL archive at 4 GB without ZIP64, and the bar is
    "as much as the device can store" → >4 GB is in scope. The only ways to get
    a >4 GB zip are (a) hand-roll ZIP64 — too bug-prone for the ONLY backup, or
    (b) vendor a zip lib (fflate) — a dependency the project deliberately avoids.
    A length-prefixed container sidesteps BOTH: 64-bit lengths natively (no 4 GB
    wall), and it is so simple it is hard to get wrong — the real data-safety win
    on this path is MINIMAL BUG SURFACE, not external tooling.

   Why this format:
    • 64-bit lengths → no 4 GB / ZIP64 problem; handles whatever the device holds.
    • Dead simple → smallest possible bug surface on the only-backup path; fully
      auditable; recoverable from this spec alone (a ~20-line script).
    • No dependency (honors the lean-deps + "no added risk" posture).
    • No base64 → ~25% smaller + no encode/decode CPU + no 1.33x heap.
    • Streams trivially BOTH ways: random-access via Blob.slice on web (the codec
      reads each frame as a lazy slice — bounded), and sequentially on Android
      (native reads magic→len→manifest→[len+bytes]* straight off the SAF stream;
      the JS codec is NOT used on Android — native writes/reads the same frames).
    • Integrity: each frame's length is cross-checked against the manifest's
      declared `size`; a mismatch or truncation throws (corruption is caught).
   Tradeoff accepted: the file is not openable by a generic zip tool. For a
   re-import-only backup whose owner has the source repo (the format spec lives
   here), that is acceptable; the simplicity/safety win dominates.

   STATUS: P1 LANDED — src/utils/backup-container.js (writeContainer /
   readContainer / isContainerMagic / encode|decodeUint64BE) + 14 exhaustive
   round-trip tests (byte-exact binary, the >4 GB length boundary, multi-chunk,
   unicode, truncation/size-mismatch/bad-magic/bad-JSON guards). The codec is
   web-side; Android frames natively in P3.

───────────────────────────────────────────────────────────────────────────
 I/O DESIGN — per platform
───────────────────────────────────────────────────────────────────────────
 WEB (PWA, modern browser — Chrome/Edge are the realistic target):
  • Export: File System Access API — showSaveFilePicker() → createWritable()
    gives a streaming WritableStream. Write the zip incrementally: manifest
    entry, then per media id read the blob via blob.stream() and pipe its bytes
    into the entry. Peak memory ≈ one blob. FALLBACK (Firefox/Safari, no FS
    Access API): assemble the zip as a Blob from PARTS (new Blob([...header,
    blobRef, ...]) references the IDB blobs lazily, not one buffer) → download.
    The fallback is heavier but still avoids base64; the FS Access API path is
    the GB-scale one.
  • Import: showOpenFilePicker() (or <input type=file>) → readContainer(file) →
    manifest → apply stores; each media frame is a lazy Blob.slice → IDB. Peak ≈
    one frame. Replaces readAsDataURL + JSON.parse-of-the-whole-file.

 ANDROID (APK / WebView — native owns the file I/O + the framing; JS feeds it):
  Two WV69 facts shape this: (1) the JS↔native bridge can't carry GBs in one
  string arg, and (2) WebView-69 lacks Blob.arrayBuffer/.stream, so the web codec
  above cannot run there — JS must read blob bytes via FileReader per ≤1 MB slice
  (rule 5). So native RE-IMPLEMENTS THE SAME framed format in Kotlin (trivial —
  magic + manifest + length-prefixed frames). This dual-impl is precisely why the
  format is dumb-simple: a ZIP would be unsafe to dual-impl / need a dep. New
  AppInterface @JavascriptInterface methods + JsEvent callbacks; native side in
  StorageManager + the existing SAF pickers:
  • Export: JS picks the SAF destination → native opens the OutputStream, writes
    magic + manifest. Then per media id JS reads the blob via FileReader (≤1 MB
    slices) and hands native each slice as base64:
      exportBegin(manifestJson)     → native writes magic + manifest frame
      exportWriteBlob(id, size)     → native writes the next frame's length
      exportBlobChunk(base64Chunk)  → native decodes + writes raw bytes
      exportEnd()                   → native flushes + closes
    Only one ≤1 MB slice (~1.3 MB base64) crosses the bridge at a time.
  • Import: JS picks the file → native opens the InputStream, reads magic +
    manifest, returns it (one small call), then streams each frame to JS:
      importBegin()                 → native opens, returns manifestJson
      importNextBlob()              → native advances to the next frame
      importReadChunk(n)            → native returns ≤n bytes base64 ("" at end)
    JS reassembles each blob (bounded) and puts it to JournalMediaStore.
  Byte-compatibility web↔Android (export on one, import on the other) IS the
  format's contract, locked by the P3 cross-platform round-trip tests.
  NB: per-blob is bounded (audio + images are MB-scale; the app has no video).
  Only the COUNT of blobs is unbounded — which streaming handles.

 BACK-COMPAT (must not strand existing backups):
  • Detect format on import by the first bytes: "VOTBACK1" → v3 streaming path;
    a leading '{' → the existing v1/v2 base64-JSON path (keep applyImportPayload
    + JSON.parse for those — they are small + old). isContainerMagic() already
    does this sniff.
  • Export always writes v3 going forward.
  • Old app versions can't read v3 — acceptable (forward-only); document it.

 CAPS:
  • Remove WEB_MAX_IMPORT_BYTES / MAX_IMPORT_SIZE / DEFAULT_MEDIA_LIMIT_BYTES
    as hard refusals once streaming lands. Replace with a SOFT, advisory
    free-space check (navigator.storage.estimate / StatFs) that WARNS but does
    not block — the user owns their device's limits.
  • S5's aggregate import cap: drop it on the v3 streaming path (memory is now
    bounded per-blob); keep it only as a guard on the legacy base64 path.

 VERIFICATION BAR (non-negotiable for the only-backup path):
  • Byte-exact media round-trip (sha256 each blob in == out) at >1 GB synthetic.
  • Cross-platform: a backup made on web imports on Android and vice versa.
  • Peak-memory assertion: instrument the import/export to prove memory stays
    ~one-blob, not whole-payload.
  • No data loss on a mid-stream failure (partial import must not wipe existing
    data — write to a staging area / all-or-nothing per store, like today's
    durability barrier).
